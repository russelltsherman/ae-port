# Rust slice hint — Slice 0005: Integer id parsing and 400 validation error

Target: **rust** · Stack: **axum 0.7 + tokio + serde/serde_json** (established in `dependency-map.md:24-31`).

This hint is PRESCRIPTIVE. It specifies exactly ONE way to build this slice in Rust. Do not
substitute alternatives. Every behavior below is grounded in the Go source
(`src/examples/seeds-go/server.go`); citations are `path:line`.

---

## 0. What this slice is

Inside the `GET /items/{id}` axum handler, parse the `{id}` path segment as an integer. On parse
failure, return HTTP **400** with the exact JSON body `{"error":"id must be an integer"}` and
`Content-Type: application/json`, and return from the handler immediately (no store lookup). On
success, the parsed integer flows to the store lookup (lookup/404/200 owned by other slices).

Go source for the guard (`server.go:46-52`):

```go
mux.HandleFunc("GET /items/{id}", func(w http.ResponseWriter, r *http.Request) {
    id, err := strconv.Atoi(r.PathValue("id"))   // server.go:47
    if err != nil {                               // server.go:48
        writeError(w, http.StatusBadRequest, "id must be an integer")  // server.go:49
        return                                    // server.go:50
    }
    it, err := store.Get(id)                      // server.go:52
    // ... lookup/404/200 owned by other slices
})
```

## 1. Hard rules (non-negotiable)

1. **Parse type is `i64`, NOT `i32` / `usize`.** Go `int` is 64-bit on the targets in scope and
   `strconv.Atoi` returns `int` (`dependency-map.md` H8 / `porting-hazards.md:166-176`). Use
   `.parse::<i64>()`. The `id`, `next_id`, and store key must all be `i64` for consistency with
   sibling slices.

2. **Extract `Path<String>` and parse manually. Do NOT use `Path<i64>`.** axum's `Path<i64>`
   auto-extraction produces its OWN 400 with a DIFFERENT body (a plain-text rejection like
   `Cannot parse "abc" to a i64`), which violates the exact-body contract. You MUST own the parse
   so you own the error response. Grounded: `porting-hazards.md:178-203` (H9),
   `porting-hazards.md:194-203`, and `dependency-map.md:78-83`.

3. **The error message string MUST be EXACTLY** `id must be an integer` — byte-for-byte. No
   capitalization, no trailing period, no rephrasing (`server.go:49`).

4. **The error body MUST serialize to EXACTLY** `{"error":"id must be an integer"}`. In Go this is
   `map[string]string{"error": msg}` (`server.go:83`). Reproduce with the shared error writer (see
   §3). `serde_json` with a single key emits exactly `{"error":"..."}` — single key, so no field
   ordering concerns.

5. **`Content-Type` MUST be `application/json`** on the 400 response (Go: `server.go:77` via
   `writeError`). This is provided by the shared `write_json` helper (§3); do not set it by hand.

6. **Return immediately after writing the error.** No store lookup may run on the failure path
   (`server.go:50` `return`). In Rust this is `return write_json(...)` inside the match arm.

7. **No trimming, no special-casing.** `strconv.Atoi` does NOT trim whitespace and accepts only an
   optionally-signed base-10 integer. Rust `i64::parse` has the SAME behavior. A single
   `.parse::<i64>()` + error guard reproduces every edge case automatically. Do NOT add
   `.trim()`, radix handling, or any extra validation (`server.go:42`,
   `0005...md` §2 edge-case table, `porting-hazards.md:188-192`).

## 2. Edge-case behavior (driven by `i64::parse`, which matches `strconv.Atoi`)

These MUST all yield 400 `{"error":"id must be an integer"}` — verified equivalent between
`strconv.Atoi` and `i64::parse` in `porting-hazards.md:188-192`:

| Input `{id}` | Result | Reason |
|---|---|---|
| `"abc"` | 400 | non-numeric (the only Go-test-covered case, `server_test.go:87`) |
| `""` (empty) | 400 | empty parses to error |
| `"1.5"` | 400 | float syntax not accepted |
| `"0x1"` | 400 | non-base-10 prefix not accepted |
| `" 1"` / `"1 "` | 400 | leading/trailing whitespace NOT trimmed by either Go or Rust |
| `"99999999999999999999999999999"` (overflow) | 400 | exceeds `i64` range → parse error |
| `"+"` alone | 400 | both Go and Rust error |

These MUST all SUCCEED (parse OK, NO error response, control flows to store lookup):

| Input `{id}` | Parsed `i64` | Reason |
|---|---|---|
| `"+5"` | `5` | leading `+` accepted by both |
| `"-5"` | `-5` | leading `-` accepted |
| `"007"` | `7` | leading zeros accepted |
| `"0"` | `0` | accepted |

> Do NOT special-case any of these. One `.parse::<i64>()` + the `Err` arm produces all rows above.

> Note on the empty case: a request to `/items/` is matched by the list route `GET /items`, not by
> `/items/{id}`, so an empty segment does not normally reach this handler. Implement the guard so
> that IF an empty string ever reaches the parse it yields 400 (it does automatically). Same
> caveat as `0005...md` §AT-4.

## 3. Dependency: shared error writer (from the SHARED-response-helpers slice)

This slice CALLS the shared writer; it MUST NOT redefine it. The established signature
(`porting-hazards.md:88-96`, H4) is:

```rust
fn write_json<T: Serialize>(status: StatusCode, v: T) -> Response
```

The error envelope in Go is `map[string]string{"error": msg}` (`server.go:83`). Reproduce the
error body with one of these (pick the `json!` form for this slice — fewest moving parts):

```rust
use serde_json::json;
write_json(StatusCode::BAD_REQUEST, json!({ "error": "id must be an integer" }))
```

`write_json` is responsible for setting `Content-Type: application/json` and the status code
(mirrors Go `writeJSON` at `server.go:76-78`). Treat it as a black box: depend only on
status + `{"error":msg}` body + JSON content type.

If your SHARED slice instead exposes a convenience `write_error(status, msg) -> Response` wrapper
(mirroring Go `writeError`, `server.go:82-84`), prefer calling that:

```rust
write_error(StatusCode::BAD_REQUEST, "id must be an integer")
```

Use whichever the SHARED slice actually defined. Do NOT define either here.

## 4. Exact handler code for THIS slice

Handler signature (axum 0.7, manual-parse pattern). `Path<String>` because we parse ourselves:

```rust
use axum::{extract::{Path, State}, http::StatusCode, response::Response};
use serde_json::json;
use std::sync::Arc;

async fn get_one(
    State(store): State<Arc<Store>>,
    Path(raw_id): Path<String>,
) -> Response {
    // --- THIS SLICE: integer parse + 400 guard (mirrors server.go:47-50) ---
    let id: i64 = match raw_id.parse() {
        Ok(n) => n,
        Err(_) => {
            return write_json(
                StatusCode::BAD_REQUEST,
                json!({ "error": "id must be an integer" }),
            );
        }
    };

    // --- success path: id (i64) flows to the store lookup (server.go:52) ---
    // Lookup / 404 / 200 serialization owned by other slices. Example shape:
    // match store.get(id) { ... }
    todo!("store.get(id) lookup owned by the lookup slice")
}
```

Notes:
- `let id: i64` — the explicit type annotation is what selects `parse::<i64>()`. Do not omit it
  or Rust cannot infer the target type.
- `Err(_)` — discard the parse error; the message is a fixed literal, never `err.to_string()`.
  (Go also uses a fixed literal here, `server.go:49`, NOT `err.Error()`.)
- The `return` inside the `Err` arm is mandatory — it guarantees no store lookup on failure
  (matches `server.go:50`).
- `State<Arc<Store>>` extractor matches the store-wrapping decision in `dependency-map.md:187-190`
  (`Arc<Store>` containing a `std::sync::Mutex`). The exact store API is owned by other slices;
  only the parse guard and the `i64` value handed downstream belong to THIS slice.

The DELETE handler (`server.go:60-64`) has the IDENTICAL parse guard. That is a different slice;
do not build it here, but the parse logic is the same shape.

## 6. Tests to ADD (TDD obligation — Go coverage is THIN)

The Go suite tests ONLY `GET /items/abc => 400` (`server_test.go:87-89`). It does NOT assert the
JSON body, the Content-Type, or other malformed inputs (LOUD COVERAGE FLAG in `0005...md` §5).
A correct rebuild MUST implement the exact body + header anyway, and MUST add tests for them.

Use the in-process axum test harness with `tower::ServiceExt::oneshot` (the established equivalent
of Go's `httptest` + `ServeHTTP`, `dependency-map.md:237-243`). Dev-dependencies:
`tower = { version = "0.5", features = ["util"] }` and `http-body-util` (`dependency-map.md:243`).
`[UNVERIFIED]` versions — pick latest compatible.

Write ONE `#[tokio::test]` with a `for` loop over cases (Rust has no native subtests;
`dependency-map.md:234-236`). Required assertions per the slice acceptance tests
(`0005...md` §5):

- **AT-1:** `GET /items/abc` → status `400`.
- **AT-2:** status `400` AND body bytes deep-equal `{"error":"id must be an integer"}`
  (parse the body with `serde_json::from_slice::<serde_json::Value>` and compare to
  `json!({"error":"id must be an integer"})`).
- **AT-3:** response header `Content-Type` == `application/json`.
- **AT-4:** at least `"1.5"` and `"0x1"` (and ideally `"%201"` for `" 1"`) → 400 with same body.
- **AT-5:** a valid integer (`GET /items/1` after creating an item) → status **!= 400**
  (do NOT assert exactly 200; the 200/404 outcome is owned by the lookup slice).

Test skeleton:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use axum::body::Body;
    use axum::http::{Request, StatusCode};
    use http_body_util::BodyExt;
    use tower::ServiceExt; // for `oneshot`

    fn app() -> axum::Router {
        new_server(Arc::new(Store::new())) // constructor owned by other slices
    }

    #[tokio::test]
    async fn non_integer_id_returns_400() {
        for raw in ["abc", "1.5", "0x1", "%201"] {
            let resp = app()
                .oneshot(
                    Request::builder()
                        .method("GET")
                        .uri(format!("/items/{raw}"))
                        .body(Body::empty())
                        .unwrap(),
                )
                .await
                .unwrap();

            assert_eq!(resp.status(), StatusCode::BAD_REQUEST, "input {raw:?}");
            assert_eq!(
                resp.headers()
                    .get("content-type")
                    .and_then(|v| v.to_str().ok()),
                Some("application/json"),
                "input {raw:?}",
            );
            let bytes = resp.into_body().collect().await.unwrap().to_bytes();
            let body: serde_json::Value = serde_json::from_slice(&bytes).unwrap();
            assert_eq!(body, serde_json::json!({ "error": "id must be an integer" }));
        }
    }
}
```

Adjust constructor / `Store::new` names to whatever the route + store slices actually defined.

## 7. Anti-patterns — do NOT do these

- ❌ `Path<i64>` (wrong 400 body — `porting-hazards.md:194-203`).
- ❌ `Path<i32>` or `usize` (wrong width — `porting-hazards.md:166-176`).
- ❌ `.unwrap()` / `.expect()` on `parse` (a panic aborts the request; `porting-hazards.md:420-425`).
- ❌ `raw_id.trim().parse()` (Go does not trim; `server.go:42`).
- ❌ Using `err.to_string()` as the message (Go uses a fixed literal, `server.go:49`).
- ❌ Redefining `write_json` / `write_error` here (owned by SHARED slice, §3).
- ❌ Re-registering the route (owned by EPIC-002-S1, §5).
- ❌ Implementing `store.get` / the 404 / 200 path (other slices, `0005...md` §4).
