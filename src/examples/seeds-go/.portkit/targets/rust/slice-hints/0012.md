# Rust slice hint ‚Äî EPIC-004-S2 (build #12): `Store::delete` returns `NotFound` for an absent id

**Scope:** ONLY the absent-id branch of `Store::delete`. Given an id that is NOT a key in the
store, return the `NotFound` error variant and leave the store completely unchanged. The
present-id (success) path is slice S1 ‚Äî do NOT redefine or change it here. This slice adds the
absence guard in front of the S1 removal.

## 3. EXACT behavior to reproduce (absent-id branch)

Mirror the Go semantics at `store.go:77-83` precisely:

1. **Acquire the lock first.** Lock the `Mutex` before inspecting the map. The guard is held
   for the whole method body (the Rust equivalent of `s.mu.Lock(); defer s.mu.Unlock()` at
   store.go:77-78). Releasing happens automatically at scope end.

2. **Presence check via the map, value discarded.** Go does
   `if _, ok := s.items[id]; !ok { return ErrNotFound }` (store.go:79-80) ‚Äî it discards the
   value and tests only presence. In Rust, test membership without removing:
   `if !inner.items.contains_key(&id) { return Err(AppError::NotFound); }`.

3. **Return the bare error variant.** On absence, return `Err(AppError::NotFound)` directly ‚Äî
   do NOT wrap, annotate, or construct a new error. Grounded in store.go:80 returning the bare
   sentinel; callers compare with `matches!(..., AppError::NotFound)` (dependency-map.md:120).

4. **No mutation on the absent path.** The early return MUST happen before any removal ‚Äî exactly
   as Go's `return ErrNotFound` at store.go:80 precedes `delete(s.items, id)` at store.go:82.
   On an absent-id call: nothing is inserted, nothing is removed, and no id counter is touched
   (`nextID`/`next_id` is only modified in `create`, store.go:48, never in delete). The store is
   byte-for-byte unchanged.
   - [PARTIALLY UNVERIFIED] The Go source test does not re-count the store after a failed delete
     to prove non-mutation (slice doc store-delete ¬ß1.4 note at .md:50). Non-mutation is grounded
     in source control flow (store.go:79-82) only. Include the strengthening test AT-4 below.

5. **Mutual exclusivity with S1.** Exactly one of `Ok(())` / `Err(AppError::NotFound)` is
   returned per call: `Ok(())` for a present id (S1, store.go:82-83), `Err(AppError::NotFound)`
   for an absent id (this slice, store.go:79-80). No other outcome exists.

6. **No panic.** `HashMap::contains_key` on an absent key is safe and never panics, matching
   Go's safe comma-ok read on a non-nil map (slice doc ¬ß1.6 / store.go:53 reference). The map is
   always initialized (S1 constructor, store.go:28).

## 4. Reference implementation (the guard added in front of S1)

Insert the absence guard as the FIRST statement after taking the lock, before the S1 removal.

```rust
pub fn delete(&self, id: i64) -> Result<(), AppError> {
    let mut inner = self.inner.lock().unwrap();          // S1 scaffolding: Mutex guard
    if !inner.items.contains_key(&id) {                  // THIS SLICE: absence guard
        return Err(AppError::NotFound);                  // store.go:79-80
    }
    inner.items.remove(&id);                             // S1 success path (store.go:82)
    Ok(())                                               // S1 success path (store.go:83)
}
```

Notes:
- Field name `self.inner` is illustrative ‚Äî use whatever the S1 `Store` actually named its
  `Mutex<StoreInner>` field. Do not rename it.
- `.lock().unwrap()` mirrors Go's unconditional lock; a poisoned mutex is not a modeled case
  here (no panic occurs while the lock is held in this code).
- The success path (`remove` + `Ok(())`) belongs to S1 and is shown only for placement context;
  do not modify it.

### `AppError::NotFound` message (must already exist from the error-type slice)

The string MUST be exactly `item not found` ‚Äî no prefix, suffix, or punctuation
