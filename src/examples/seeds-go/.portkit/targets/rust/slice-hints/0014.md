# Rust SLICE-HINT — #14: DELETE /items/{id} returns 204 No Content on successful delete

Target: **rust**. Slice: EPIC-004-S4. dependsOn: EPIC-004-S1 (`Store::delete`), EPIC-004-S3 (DELETE route + integer-parse/400 branch).

This is PRESCRIPTIVE. Use exactly the libraries, signatures, and idioms below. Do not substitute alternatives.

This slice owns ONE thing: the fall-through after `store.delete(id)` returns `Ok(())` (the item existed and was removed), which sends HTTP `204 No Content` with an EMPTY body. The 400 (non-integer id) and 404 (missing id) branches of the same handler belong to other slices but MUST be present and ordered correctly for this success path to be reachable.

Source behavior is grounded to the Go code: handler `server.go:60-71`, store method `store.go:76-84`, test `server_test.go:110-123`.

## 2. Exact decision logic for the DELETE handler (reproduce in this order)

Grounded ordering from `server.go:61-70`. The handler does, in order:

1. Read raw `{id}` segment, parse to `i64`. On parse error → `400 {"error":"id must be an integer"}` and return. (Owned by EPIC-004-S3 / porting-hazards H9; MUST be present and FIRST so the success path is only reachable for integer ids — `server.go:61-64`.)
2. Call `store.delete(id)` EXACTLY ONCE and test its result. If it returns the not-found error → `404 {"error":"item not found"}` and return. (Owned by EPIC-004-S5; the single `delete` call here is what performs the delete on the success path — `server.go:66-68`.)
3. **THIS SLICE:** otherwise (delete returned `Ok(())`) → respond `204 No Content`, EMPTY body, NO `Content-Type` header set by the handler. Return. (`server.go:70-71`.)

Wrong order breaks correctness: parse-400 MUST precede the delete; the 404 test MUST precede the 204 fall-through (slice doc §2 "Ordering", `server.go:61-70`).

## 3. `Store::delete` contract relied upon (from EPIC-004-S1 — DO NOT reimplement here)

Grounded to `store.go:76-84`. Use this exact Rust signature:

```rust
// returns Ok(()) when the id existed and was removed; Err(AppError::NotFound) when absent.
pub fn delete(&self, id: i64) -> Result<(), AppError>;
```

- Returns `Ok(())` when the id existed and was removed (`store.go:82-83`: `delete(s.items, id); return nil`).
- Returns `Err(AppError::NotFound)` when the id was absent (`store.go:79-80`: `if _, ok := ...; !ok { return ErrNotFound }`).
- Thread-safe: locks the inner mutex for the whole method (`store.go:77-78`); the Rust impl takes `self.inner.lock().unwrap()` (porting-hazards H2). The `delete` body removes the key only if present, so a second delete of the same id is `Err(NotFound)` — this is the idempotency boundary (slice doc §2; `store.go:79-80`).
- `AppError::NotFound`'s `Display` MUST be exactly `"item not found"` (`store.go:13`; KERNEL sentinel glossary). Returning that error maps to HTTP 404.

Treat `delete` as opaque with the behavior above. Do not call it twice; the single call both removes and reports (slice doc §2 "Exactly one Delete call"; `server.go:66`).

## 4. EXACT handler implementation (this slice's code)

axum handler. Use `Path<String>` (NOT `Path<i64>`) and parse manually so YOU control the 400 body (porting-hazards H9/H11). Shared store via `State<Arc<Store>>`.

```rust
use std::sync::Arc;
use axum::{
    extract::{Path, State},
    http::StatusCode,
    response::{IntoResponse, Response},
    Json,
};
use serde_json::json;

async fn delete_item(
    State(store): State<Arc<Store>>,
    Path(raw_id): Path<String>,
) -> Response {
    // Step 1 (EPIC-004-S3): integer-parse / 400. Present + FIRST. (server.go:61-64)
    let id: i64 = match raw_id.parse() {
        Ok(v) => v,
        Err(_) => {
            return (
                StatusCode::BAD_REQUEST,
                Json(json!({ "error": "id must be an integer" })),
            )
                .into_response();
        }
    };

    // Step 2 (EPIC-004-S5 arm; THIS SLICE makes the single delete call). (server.go:66-68)
    if let Err(AppError::NotFound) = store.delete(id) {
        return (
            StatusCode::NOT_FOUND,
            Json(json!({ "error": "item not found" })),
        )
            .into_response();
    }

    // Step 3 — THIS SLICE'S OUTPUT: 204, empty body, no Content-Type. (server.go:70-71)
    StatusCode::NO_CONTENT.into_response()
}
```

Notes (grounded, prescriptive):

- The success response is `StatusCode::NO_CONTENT.into_response()` — returning a bare `StatusCode` in axum yields an EMPTY body with NO `Content-Type` header. This mirrors `server.go:70`, which calls only `w.WriteHeader(http.StatusNoContent)` and writes no body and sets no header (slice doc §2 "Content-Type header: NOT set"; porting-hazards H17). Do NOT wrap it in `Json(...)`, do NOT write any body, do NOT set any header.
- `store.delete(id)` is called exactly once, inside the `if let` (`server.go:66`). Do not duplicate it.
- The 404 message is the literal sentinel string `"item not found"` (Go passes `ErrNotFound.Error()`, not a wrapped error's message — `server.go:67`). Hard-coding it here is fine; it equals `AppError::NotFound`'s `Display`. If `store.delete` ever returns a different `AppError` variant (it cannot per the contract), this `if let` only catches `NotFound` and would fall through to 204 — that is acceptable because the contract guarantees only `Ok(())` or `Err(NotFound)` (`store.go:79-83`).
- Route registration (shared across slices; from porting-hazards H16): `.route("/items/{id}", get(get_item).delete(delete_item))` on the `Router`, with `.with_state(store)`.

Do NOT use `.unwrap()`/`.expect()`/panic on the request path (porting-hazards H19). The only acceptable `.unwrap()` is on `Mutex::lock()` inside `store.delete` (which lives in EPIC-004-S1).

## 5. Side effect

The item with `id` is removed from the store (`store.go:82`). A subsequent operation on the same id behaves as not-found. Ids are sequential, start at 1, and are NEVER reused — delete does not decrement `next_id` (`store.go:28,48,82`; KERNEL "Domain vocabulary: id"). After deleting id 1, the next create gets a fresh id, not 1.
