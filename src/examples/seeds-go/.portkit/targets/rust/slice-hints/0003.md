## Request body type (this slice DEFINES this — it is local to the handler)

Port of the Go anonymous struct `struct { Name string \`json:"name"\` }` (`server.go:22-24`).
Go's decoder reads only the `name` key and ignores any other JSON fields (`server.go:22-25`).
serde_json has the same default behavior (unknown fields ignored unless
`#[serde(deny_unknown_fields)]` is set — do NOT set it).

```rust
#[derive(serde::Deserialize)]
struct CreateBody {
    name: String,
}
```

- Field name `name` matches the Go tag `json:"name"` (`server.go:23`) with no rename needed.
- Do NOT add `#[serde(deny_unknown_fields)]` — Go ignores extra fields, so must this.
- The field is `String` (required). If the JSON `name` is present but the wrong type (e.g. a
  number), `from_slice` returns `Err`, which maps to outcome #1 (400 "invalid JSON body"),
  matching Go's `Decode` returning an error for a wrong-typed value (`server.go:25-27`).

## Handler body — exact translation of `server.go:25-39`

Reproduce the Go logic statement-for-statement. The control flow and outcome ordering are part of
the HTTP contract (slice doc "Outputs — exhaustive outcome table"; hazard H20).

```rust
async fn create_item(
    axum::extract::State(store): axum::extract::State<std::sync::Arc<Store>>,
    body: axum::body::Bytes,
) -> axum::response::Response {
    // 1. Decode the JSON body manually (port of server.go:25-28).
    let body: CreateBody = match serde_json::from_slice(&body) {
        Ok(b) => b,
        Err(_) => {
            return write_error(axum::http::StatusCode::BAD_REQUEST, "invalid JSON body");
        }
    };

    // 2. Call the store (port of server.go:29). Pass body.name exactly as decoded;
    //    the store trims/validates it (store.go:34,37; hazard H10).
    match store.create(&body.name) {
        // Case A: validation errors -> 400 with the error's message (server.go:31-32).
        Err(e @ (AppError::NameRequired | AppError::NameTooLong)) => {
            write_error(axum::http::StatusCode::BAD_REQUEST, &e.to_string())
        }
        // Case B: duplicate -> 409 with the error's message (server.go:33-34).
        Err(e @ AppError::Duplicate) => {
            write_error(axum::http::StatusCode::CONFLICT, &e.to_string())
        }
        // Case C: any other error -> 500 "internal error" (server.go:35-36).
        // Defensive catch-all; unreachable with the in-memory store but REQUIRED for parity.
        Err(_) => {
            write_error(axum::http::StatusCode::INTERNAL_SERVER_ERROR, "internal error")
        }
        // Case D: success -> 201 + the created Item as JSON (server.go:37-38).
        Ok(it) => write_json(axum::http::StatusCode::CREATED, it),
    }
}
```

### Exact-behavior requirements (do not deviate)

1. **Decode happens BEFORE `create` is ever called** (`server.go:25-29`; slice doc "Ordering
   guarantee"). On a decode error, return IMMEDIATELY without calling `store.create`
   (`server.go:26-27`). The early `return` in the `Err` arm above enforces this.

2. **Empty body → outcome #1 (400 "invalid JSON body"), NOT outcome #2.** Go's `Decode` on an
   empty reader returns `io.EOF`, a non-nil error (`server.go:25-27`; slice doc line 54).
   `serde_json::from_slice(&[])` on empty bytes returns `Err` ("EOF while parsing a value"), so
   it naturally maps to outcome #1. Do not special-case it. The test harness sends a `nil`/empty
   body for the empty-body case (`server_test.go:18-19`).

3. **The two 400 outcomes differ only in message** (slice doc "Ordering guarantee"):
   - Decode failure → `{"error":"invalid JSON body"}` (a literal string you pass).
   - Validation failure → `{"error":"<err message>"}`, e.g. `{"error":"name is required"}` — the
     message comes from `AppError::to_string()` (Go's `err.Error()`, `server.go:32`).
   Do NOT hardcode the validation message; always use `e.to_string()` so it echoes the sentinel's
   exact text (`server.go:31-32`; dependency-map.md item 4).

4. **Error matching uses the enum variants directly** — the Rust analog of Go's `errors.Is`
   (`server.go:31,33`; hazard convention). Pattern-matching `AppError::NameRequired` etc. is the
   correct port of `errors.Is(err, ErrNameRequired)`. Match the FIRST-WINS ordering exactly:
   NameRequired/NameTooLong → Duplicate → other → Ok. (If `AppError` is a non-`Copy` enum and the
   `@`-binding-with-or-pattern form fights the borrow checker, use a `match e { ... }` inside an
   `Err(e) =>` arm instead — see "Alternative match form" below. The ORDER and the status mapping
   must not change.)

5. **`name` is passed through verbatim** to `store.create` (`server.go:29`: `body.Name` passed
   as-is). The store does the trimming and validation (`store.go:34,37`; hazard H10). Do NOT trim
   or validate `body.name` in the handler.

6. **Success status is exactly 201 (`StatusCode::CREATED`), not 200** (`server.go:38`; slice doc
   line 116). The success body is the bare `Item` encoded as JSON via `write_json` (no `{"error"}`
   envelope — that envelope is error-only) (`server.go:38,76-79`; hazard H4).

7. **The 500 branch MUST exist** even though it is unreachable with the kernel's in-memory store
   (`store.create` only ever returns the three known sentinels — `store.go:34-49`). It is a
   defensive catch-all in the source (`server.go:35-36`; slice doc lines 65-69, outcome #4) and is
   required for parity. Map any other/unexpected error to 500 `{"error":"internal error"}` (a
   literal string, NOT `err.to_string()` — Go uses the fixed string `"internal error"` here,
   `server.go:36`).

### Alternative match form (if borrow checker rejects `@`-or-patterns)

Functionally identical; same ordering and statuses:

```rust
match store.create(&body.name) {
    Ok(it) => write_json(StatusCode::CREATED, it),
    Err(e) => match e {
        AppError::NameRequired | AppError::NameTooLong => {
            write_error(StatusCode::BAD_REQUEST, &e.to_string())
        }
        AppError::Duplicate => write_error(StatusCode::CONFLICT, &e.to_string()),
        _ => write_error(StatusCode::INTERNAL_SERVER_ERROR, "internal error"),
    },
}
```

Note: the inner `match e { ... }` borrows `e` for `e.to_string()`; bind by reference
(`Err(ref e)` or rely on default match ergonomics) if the compiler complains about a move.

## Hazard: DO NOT use axum's `Json<T>` extractor (hazard H11)

If you take `Json<CreateBody>` as a handler argument instead of raw `Bytes`, axum rejects a
malformed body BEFORE your handler runs and returns axum's DEFAULT plain-text 400 (e.g.
`Failed to parse the request body as JSON: ...`) — NOT `{"error":"invalid JSON body"}`. That
breaks the contract verified by `server_test.go:50` (`{not json` → 400 with the specific body).

MANDATORY fix: take `body: Bytes`, call `serde_json::from_slice::<CreateBody>(&body)`, and on
`Err` return `write_error(StatusCode::BAD_REQUEST, "invalid JSON body")` yourself. This is the
only way to reproduce Go's exact decode-failure 400 body (`server.go:25-27`).

## Route registration

Port of `mux.HandleFunc("POST /items", ...)` (`server.go:21`). Register on the `axum::Router`
inside the router-constructor (the port of `NewServer`, `server.go:18-74`):

```rust
use axum::routing::post;

Router::new()
    .route("/items", post(create_item))   // POST /items -> this handler
    // ... other routes added by their own slices ...
    .with_state(store)                     // store: Arc<Store>
```

- `post(create_item)` makes the route dispatch ONLY `POST` requests for path `/items` to
  `create_item` — the analog of Go's `"POST /items"` method+path pattern (`server.go:21`; hazard
  H16). A `GET`/`DELETE` on `/items` is handled by the OTHER slices' registrations chained onto
  the same `.route("/items", ...)` (e.g. `post(create_item).get(list_items)`); this slice
  contributes only the `post(...)` part. Do not register the other methods here.
- `.with_state(store)` injects the `Arc<Store>` that the `State` extractor pulls out (hazards
  H2/H3; `server.go:18` `*Store`).
- Path is the exact literal `/items` (no `{id}`), matching `server.go:21`.
