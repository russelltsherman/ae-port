## 2. The error enum (exact)

Model the sentinels as ONE enum. For slice #1 declare it with the two variants used here. The
`#[error("…")]` strings MUST be byte-for-byte identical to the Go messages, because later slices
return them verbatim to HTTP clients via `err.Error()`.

```rust
use thiserror::Error;

#[derive(Debug, Clone, PartialEq, Eq, Error)]
pub enum AppError {
    #[error("name is required")]
    NameRequired,
    #[error("name must be at most 100 characters")]
    NameTooLong,
    // NotFound / Duplicate added by later slices — same enum.
}
```

Exact strings (do not paraphrase, do not change capitalization or punctuation):
- `NameRequired` → `"name is required"` (`item.go:22`).
- `NameTooLong`  → `"name must be at most 100 characters"` (`item.go:23`).

Why `PartialEq, Eq`: Go matches these sentinels with `errors.Is` (`store_test.go:24`); the Rust
equivalent in tests is `assert_eq!`/`matches!`, which needs `PartialEq` (`dependency-map.md:120`).
Why `Clone`: later store slices return `Item` zero-value-free and may clone errors; harmless here.
Derive `Error` (thiserror) to get the `Display` strings above and a `std::error::Error` impl.

## 4. The function (exact)

```rust
pub fn validate_name(name: &str) -> Result<(), AppError> {
    let trimmed = name.trim();
    if trimmed.is_empty() {
        return Err(AppError::NameRequired);
    }
    if trimmed.chars().count() > MAX_NAME_LEN {
        return Err(AppError::NameTooLong);
    }
    Ok(())
}
```

### Signature mapping (Go → Rust)
- Go `func ValidateName(name string) error` (`item.go:28`) → Rust
  `pub fn validate_name(name: &str) -> Result<(), AppError>`.
  - Take `&str`, NOT `String`: the function only reads the input, never owns it (`item.go:28-37`
    reads `name`, no mutation — neutral doc §2 "No mutation / no side effects").
  - Return `Result<(), AppError>`: Go's `nil` success ↔ `Ok(())`; Go's returned sentinel ↔
    `Err(AppError::…)`. There is no value to return on success.
- Function name: snake_case `validate_name` (Rust convention; Go exported `ValidateName`).

### Algorithm — implement in EXACTLY this order (`item.go:29-36`)
1. `let trimmed = name.trim();` — mirrors `trimmed := strings.TrimSpace(name)` (`item.go:29`).
   ALL subsequent checks use `trimmed`, NEVER `name` (neutral doc §1 step 2).
2. Empty check FIRST: `if trimmed.is_empty() { return Err(AppError::NameRequired); }`
   (`item.go:30-32`). This runs unconditionally before the length check (`item.go:30` precedes
   `item.go:33`).
3. Length check SECOND: `if trimmed.chars().count() > MAX_NAME_LEN { return Err(NameTooLong); }`
   (`item.go:33-35`).
4. `Ok(())` (`item.go:36`).

Ordering is load-bearing even though the two error conditions are mutually exclusive in practice
(an empty string cannot be too long) — keep the empty check first (neutral doc §2 "Ordering /
precedence"; `porting-hazards.md:210-213`).

## 5. The three non-obvious correctness rules (DO NOT get these wrong)

### 5a. Length is RUNES (code points), NOT bytes
Go uses `len([]rune(trimmed))` (`item.go:33`) — counts Unicode code points. Rust equivalent is
`trimmed.chars().count()` — `chars()` iterates Unicode scalar values (code points), the exact
analogue (`porting-hazards.md:259-273`).

- MUST use `trimmed.chars().count()`.
- MUST NOT use `trimmed.len()` (that is the BYTE length in Rust — wrong).
- MUST NOT use `.bytes().count()` or grapheme-cluster counting.
- Consequence: 100 `"é"` characters = 200 bytes but 100 code points → VALID
  (neutral doc §4 multi-byte case; `item.go:33`).

### 5b. Boundary is strictly greater-than
Use `> MAX_NAME_LEN`, NOT `>=`. Exactly 100 runes is VALID; only 101+ triggers `NameTooLong`
(`item.go:33`, `store_test.go:19-20`). `> 100` matches Go's `> MaxNameLen`.

### 5c. Trim whitespace set
Go's `strings.TrimSpace` trims leading/trailing runes where `unicode.IsSpace` is true
(`item.go:29`). Rust `str::trim()` trims leading/trailing chars where `char::is_whitespace` is
true — the correct, default equivalent (`porting-hazards.md:300-314`). Use `name.trim()`. Do not
hand-roll a custom whitespace set. The Unicode whitespace definitions differ in obscure edge
cases, but `str::trim()` is the prescribed choice and passes all existing tests; do not
over-engineer (`porting-hazards.md:312-314`).

- `is_empty()` on the trimmed `&str` is the analogue of Go's `trimmed == ""` (`item.go:30`).
  An all-whitespace input trims to `""` → `Err(NameRequired)` (neutral doc §2 "All-whitespace
  == empty"; `item.go:30`, `store_test.go:18`).

## 6. Acceptance tests (port the Go table verbatim)

Go's table-driven `TestValidateName` (`store_test.go:9-29`) matches with `errors.Is`
(`store_test.go:24`). Port it as a Rust `#[test]` with a slice of cases; compare with `==`
(the enum derives `PartialEq`). Place in a `#[cfg(test)] mod tests` in the same file.

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn validate_name_table() {
        let long = "x".repeat(MAX_NAME_LEN + 1);   // 101 → too long
        let max  = "x".repeat(MAX_NAME_LEN);        // 100 → ok
        let cases: &[(&str, &str, Result<(), AppError>)] = &[
            ("ok",              "buy milk", Ok(())),
            ("trims to ok",     "  hi  ",   Ok(())),
            ("empty",           "",         Err(AppError::NameRequired)),
            ("whitespace only", "   ",      Err(AppError::NameRequired)),
            ("too long",        &long,      Err(AppError::NameTooLong)),
            ("max length ok",   &max,       Ok(())),
        ];
        for (label, input, want) in cases {
            assert_eq!(validate_name(input), *want, "case {label}: input {input:?}");
        }
    }

    #[test]
    fn validate_name_counts_runes_not_bytes() {
        // 100 multi-byte runes (200 bytes) is VALID — proves rune counting (item.go:33).
        assert_eq!(validate_name(&"é".repeat(MAX_NAME_LEN)), Ok(()));
        // 101 multi-byte runes is too long.
        assert_eq!(
            validate_name(&"é".repeat(MAX_NAME_LEN + 1)),
            Err(AppError::NameTooLong)
        );
    }

    #[test]
    fn error_display_strings_are_exact() {
        // These strings are returned verbatim to HTTP clients in later slices.
        assert_eq!(AppError::NameRequired.to_string(), "name is required");
        assert_eq!(
            AppError::NameTooLong.to_string(),
            "name must be at most 100 characters"
        );
    }
}
```

## 8. Pitfalls (each maps to a real way this goes wrong)

- Using `name.len()` instead of `name.chars().count()` → byte count, fails multi-byte rune
  parity (`porting-hazards.md:259-273`). WRONG.
- Using `>=` instead of `>` → rejects valid 100-rune names (`item.go:33`, `store_test.go:20`).
  WRONG.
- Taking `name: String` (owning) → unnecessary allocation; the function only reads
  (`item.go:28-37`). Take `&str`.
- Trimming `name` but then validating the original `name` → must validate `trimmed`
  (`item.go:29-33`).
- Creating a per-function error type instead of the shared `AppError` enum → later slices need
  to add `NotFound`/`Duplicate` to the same enum (`dependency-map.md:116-118`,
  `porting-hazards.md:326-330`).
- Changing the message strings (e.g. "Name is required", or "100 chars") → they are returned to
  HTTP clients verbatim downstream; must be exact (`item.go:22-23`, `dependency-map.md:121-124`).
