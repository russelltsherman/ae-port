## 0. Scope of THIS slice (do not over-build)

The slice owns ONLY the 404 branch of the DELETE handler: an id that parses to a valid
integer but is NOT present in the store. The execution thread the source proves:

- `store.Delete(id)` returns `ErrNotFound` because the key is absent
  (`store.go:79-80`: `if _, ok := s.items[id]; !ok { return ErrNotFound }`).
- The handler guard `if errors.Is(store.Delete(id), ErrNotFound) {` is true
  (`server.go:66`), so it calls `writeError(w, http.StatusNotFound, ErrNotFound.Error())`
  then `return`s (`server.go:67-68`).
- Result: HTTP **404**, `Content-Type: application/json`, body `{"error":"item not found"}`
  (status `server.go:67`/`server.go:78`; content-type `server.go:77`; message `store.go:13`).

NOT in scope (owned by prerequisite slices; consume, do not re-derive):
- id parsing + 400-on-non-integer — owned by S3 (`server.go:61-64`).
- The 204 success path — owned by S4 (`server.go:70`).
- `Store::delete` internals — owned by S2 (`store.go:76-84`); this slice consumes its
  contract: `delete(id) -> Result<(), AppError>` yields `Err(AppError::NotFound)` for an

## 2. Error type the guard matches against

The Go guard matches the sentinel `ErrNotFound = errors.New("item not found")`
(`store.go:13`) via `errors.Is` (`server.go:66`). In Rust, model the store's errors as one
enum variant and match it with `matches!` (the `errors.Is` analog). Use EXACTLY this
variant and message:

```rust
// Owned by the kernel / S2 error enum. Reproduced here only for the message + variant
// this slice depends on.
#[derive(Debug)]
pub enum AppError {
    // ... other variants owned by other slices ...
    NotFound,
}

impl std::fmt::Display for AppError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            AppError::NotFound => write!(f, "item not found"), // EXACT, store.go:13
            // ... other variants ...
        }
    }
}
```

The message string MUST be exactly `item not found` (`store.go:13`). It is returned to the
HTTP client verbatim (`server.go:67` passes `ErrNotFound.Error()`; equivalently emit the
literal `"item not found"`).

NOTE (grounded source detail, reproduce exactly): the Go handler passes the package
sentinel's message `ErrNotFound.Error()` (`server.go:67`), NOT the error value returned by
`Delete`. Both resolve to the same literal `item not found` because `Delete` returns that
same sentinel (`store.go:80`). In Rust, the simplest faithful reproduction is to emit the
literal `"item not found"` in the 404 branch (see §4) rather than threading the returned
error's `Display` — either is observably identical here; prefer the literal to match the
source's intent.

## 4. The handler — EXACT signature and body

Use a free async function registered on the `/items/{id}` route with `.delete(...)`.
The handler extracts the raw id as `Path<String>` and parses it itself (do NOT use
`Path<i64>` — axum's auto-parse rejection produces a DIFFERENT 400 body; dependency map
H9/H11). The parse + 400 branch is S3's logic but appears in this same handler; reproduce
it, then add the 404 guard which is THIS slice.

```rust
use axum::{
    extract::{Path, State},
    http::StatusCode,
    response::{IntoResponse, Response},
    Json,
};
use serde_json::json;
use std::sync::Arc;

// DELETE /items/{id}  — server.go:60-71
async fn delete_item(
    State(store): State<Arc<Store>>,
    Path(raw_id): Path<String>,
) -> Response {
    // --- S3: parse id; 400 on non-integer (server.go:61-64) ---
    let id: i64 = match raw_id.parse() {
        Ok(v) => v,
        Err(_) => {
            // body EXACTLY {"error":"id must be an integer"} (server.go:63)
            return (
                StatusCode::BAD_REQUEST,
                Json(json!({ "error": "id must be an integer" })),
            )
                .into_response();
        }
    };

    // --- THIS SLICE (#15): single delete, guard on NotFound (server.go:66-68) ---
    if matches!(store.delete(id), Err(AppError::NotFound)) {
        // 404 + {"error":"item not found"} (server.go:67; message store.go:13)
        return (
            StatusCode::NOT_FOUND,
            Json(json!({ "error": "item not found" })),
        )
            .into_response();
    }

    // --- S4: success → 204, empty body (server.go:70). Early return above
    //     guarantees the absent-id path NEVER reaches this. ---
    StatusCode::NO_CONTENT.into_response()
}
```

Why this exact shape:

- **`store.delete(id)` is called exactly once** and its result is the sole input to the
  guard (`server.go:66`). Do NOT call `delete` a second time to re-check.
- **`matches!(..., Err(AppError::NotFound))` is the `errors.Is` analog** (`server.go:66`
  uses `errors.Is`, not `==`). For this enum, `matches!` on the variant is the correct,
  faithful translation.
- **Early `return` is mandatory** (`server.go:68`). It prevents the 204 fall-through
  (`server.go:70`). Without it, the absent-id path would write a body 404 AND then a 204 —
  a bug. In Rust the early `return` of the 404 `Response` enforces this structurally.
- **`Json(json!({ "error": "item not found" }))`** reproduces Go's
  `writeError`→`writeJSON` output `map[string]string{"error": msg}` →
  `{"error":"item not found"}` (`server.go:82-83`, message `store.go:13`). axum's `Json`
  responder sets `Content-Type: application/json` automatically — this satisfies the
  content-type contract from `server.go:77`.
- **Mutual exclusivity with S4**: a single valid-integer id yields EITHER 404 (this slice)
  OR 204 (S4), never both (`server.go:66` is the sole discriminator). The structure above
  guarantees this.

### AT-2 — exact 404 body AND content-type (NOT pinned by any Go test — add it)

The body is grounded in source (`server.go:67,82-83`; `store.go:13`) but NOT covered by an
existing Go test. Lock it:

```rust
#[tokio::test]
async fn delete_absent_id_body_and_content_type() {
    let app = build_router(Arc::new(Store::new()));

    let resp = app
        .clone()
        .oneshot(
            http::Request::builder()
                .method("DELETE")
                .uri("/items/999")
                .body(axum::body::Body::empty())
                .unwrap(),
        )
        .await
        .unwrap();

    assert_eq!(resp.status(), StatusCode::NOT_FOUND);

    // Content-Type: application/json (server.go:77)
    let ct = resp.headers().get(http::header::CONTENT_TYPE).unwrap();
    assert!(ct.to_str().unwrap().starts_with("application/json"));

    // body == {"error":"item not found"} (server.go:67,82-83; store.go:13)
    let bytes = http_body_util::BodyExt::collect(resp.into_body())
        .await
        .unwrap()
        .to_bytes();
    let v: serde_json::Value = serde_json::from_slice(&bytes).unwrap();
    assert_eq!(v, json!({ "error": "item not found" }));
}
```

NOTE on trailing newline: Go's `json.NewEncoder(w).Encode(v)` appends a trailing `\n`
(`server.go:79`). axum's `Json` responder does NOT append a newline. This is a known,
test-invisible byte difference — assert on the DECODED JSON value
(`serde_json::from_slice` → compare `Value`), as above, NOT on raw bytes, so the newline
difference does not matter. Do NOT byte-compare the body.

### Required AT-3 fold into AT-2

Content-type assertion is folded into AT-2 above (`server.go:77` reached via
`writeError`→`writeJSON`, `server.go:83`). `starts_with("application/json")` tolerates an
optional `; charset=utf-8` that axum's `Json` may append — the contract is the media type
`application/json`.

## 8. Pitfalls specific to this slice (do not do these)

- Do NOT use `Path<i64>` and rely on axum's auto-parse — it yields a wrong 400 body and
  steals S3's error contract (dependency map H9/H11). Parse `Path<String>` yourself.
- Do NOT call `store.delete(id)` twice (once to check, once to act). One call, result into
  the guard (`server.go:66`).
- Do NOT omit the early `return` in the 404 branch — it is what prevents the 204
  fall-through (`server.go:68` before `server.go:70`).
- Do NOT byte-compare the response body in tests (Go appends `\n` via `json.Encoder`
  `server.go:79`; axum does not). Compare decoded JSON `Value`.
- Do NOT match the error with `==`-style equality on the whole error value if your enum
  carries data; use `matches!(... , Err(AppError::NotFound))` (the `errors.Is` analog,
  `server.go:66`).
- Do NOT attach a body to the 204 success path (S4) — `server.go:70` writes no body and no
  content-type. This slice's early return keeps that path untouched.
