# Rust Slice Hint 0011 — `Store::delete` removes an existing item

Target language: **Rust**. Slice id `EPIC-004-S1`, build #11. Source root for all `path:line`
citations: `src/examples/seeds-go`.

This slice ports exactly ONE method: Go `func (s *Store) Delete(id int) error`
(`store.go:75-84`) to a Rust `Store::delete` method. Nothing else in this slice. Do NOT
re-create the `Store` type, the inner state, the mutex, or `new()` — those are KERNEL artifacts
that must already exist (see §5). If they do not exist yet, build them per the kernel doc first;
this hint only specifies `delete`.

This is PRESCRIPTIVE. Where a choice exists, exactly one option is given. Do not substitute.

---

## 1. Exact target signature

Add this method to the `impl Store` block:

```rust
/// Removes the item with the given id. Returns Ok(()) on success,
/// or Err(AppError::NotFound) if no item with that id exists.
pub fn delete(&self, id: i64) -> Result<(), AppError> {
    let mut g = self.inner.lock().unwrap();
    if !g.items.contains_key(&id) {
        return Err(AppError::NotFound);
    }
    g.items.remove(&id);
    Ok(())
}
```

That body IS the deliverable. It mirrors `store.go:76-84` line-for-line. Reasoning for every
choice is below; do not deviate.

### Why `&self` and not `&mut self`
The Go receiver is `*Store` (`store.go:76`) and mutation is guarded by an internal `sync.Mutex`
(`store.go:77-78`), NOT by exclusive ownership. The Rust port mirrors this: interior mutability
through `Mutex<StoreInner>` means `delete` takes `&self`, and the store is shared as `Arc<Store>`
across async handlers (porting-hazards.md H2, H3). Using `&mut self` would break the shared-state
model and is WRONG for this port.

### Why `i64` for `id`
Go `Item.ID` and the map key are Go `int`, which is 64-bit on the targets in scope
(`item.go:14`, `store.go:21`, dependency-map.md §6, porting-hazards.md H8). Use `i64`. Do NOT use
`i32`, `usize`, or `u64`. The map is `HashMap<i64, Item>` (porting-hazards.md H1, H2).

### Why `Result<(), AppError>` and not `Result<(), SomeOtherError>`
Go `Delete` returns a bare `error` that is either `nil` or the sentinel `ErrNotFound`
(`store.go:76,80,83`). In Rust, success carries no value, so the Ok type is `()`. The error type
is the shared kernel `enum AppError` whose `NotFound` variant maps to Go's `ErrNotFound`
(dependency-map.md §4, porting-hazards.md H15, H20). Do NOT invent a new per-method error type,
do NOT use `anyhow`/`Box<dyn Error>`, do NOT return a `String`.

## 2. Exact behavior — the body MUST do these steps in this order

Maps 1:1 to `store.go:77-83`:

1. **Acquire the lock.** `let mut g = self.inner.lock().unwrap();` — corresponds to
   `s.mu.Lock()` (`store.go:77`). The guard `g` is held for the rest of the method; it drops at
   the end of scope, which is the RAII analog of Go's `defer s.mu.Unlock()` (`store.go:78`,
   porting-hazards.md H2). `.unwrap()` on the lock is the CORRECT choice here: Go has no lock
   poisoning, and panicking on a poisoned mutex matches "do not handle it" (porting-hazards.md
   H2, H19). This is the ONLY `.unwrap()` allowed in this method.

2. **Presence check.** `if !g.items.contains_key(&id) { return Err(AppError::NotFound); }` —
   corresponds to the comma-ok lookup `if _, ok := s.items[id]; !ok { return ErrNotFound }`
   (`store.go:79-80`). On absence, return `Err(AppError::NotFound)` IMMEDIATELY and do NOT call
   `remove`. The map is NOT mutated on the not-found path (`store.go:80` returns before
   `store.go:82`).

3. **Remove the entry.** `g.items.remove(&id);` — corresponds to Go's builtin
   `delete(s.items, id)` (`store.go:82`). Discard the return value of `remove` (do not bind it).
   Reaching this line means the key is present, so exactly one entry is removed.

4. **Return success.** `Ok(())` — corresponds to `return nil` (`store.go:83`).

### Allowed micro-optimization (optional, behavior-identical)
You MAY collapse the contains-check + remove into a single `remove` call:

```rust
pub fn delete(&self, id: i64) -> Result<(), AppError> {
    let mut g = self.inner.lock().unwrap();
    match g.items.remove(&id) {
        Some(_) => Ok(()),
        None => Err(AppError::NotFound),
    }
}
```

This is observably identical: `HashMap::remove` returns `Some(old_value)` if the key was present
(success path, mirrors `store.go:82-83`) and `None` if absent (not-found path, mirrors
`store.go:79-80`), and on `None` the map is unchanged. Either form is acceptable; the two-step
form maps more visibly to the Go source. Pick ONE; do not write both.

## 3. Edge cases and guarantees (exhaustive — the rebuild MUST honor all of these)

- **Repeat delete returns `NotFound`, mutates nothing.** Deleting an id that is already absent
  returns `Err(AppError::NotFound)` and leaves the map unchanged. Grounded: the acceptance test
  deletes the same id a second time and expects not-found (`store_test.go:119-121`); Go produces
  this via `store.go:79-80`. `HashMap::remove` on an absent key returns `None` and does not
  modify the map, so the optimized form preserves this exactly.

- **No ID reuse / `next_id` untouched.** `delete` MUST NOT read or write `g.next_id`. There is no
  `next_id` reference anywhere in `store.go:76-84`. Freed ids are never reissued because `create`
  only ever increments `next_id` (`store.go:48`); `delete` does not roll it back.

- **No clock interaction.** `delete` MUST NOT touch the `now` clock field. The Go body never
  reads `s.now` (`store.go:76-84` has no `now` reference). Touch only the locked `items` map.

- **Single-key effect.** `remove(&id)` removes exactly the one keyed entry; all other ids remain
  present. This follows from `HashMap::remove` semantics, the direct analog of Go builtin
  `delete(s.items, id)` (`store.go:82`). [PARTIALLY UNVERIFIED: no source test deletes one of
  several items and re-checks the survivors; this is the documented map-delete contract, not a
  dedicated test assertion — matches the same caveat in the neutral doc §2.]

- **Error is the shared sentinel variant, compared by identity.** The not-found result is the
  kernel `AppError::NotFound` variant — the Rust equivalent of returning the bare `ErrNotFound`
  sentinel (`store.go:80`) which callers match with `errors.Is`. In Rust, compare with a `match`
  or `matches!(err, AppError::NotFound)`, NOT by comparing `Display` strings and NOT by
  allocating a new error (dependency-map.md §4, porting-hazards.md H20). Do NOT format the error
  to a string to test it.

- **Thread safety.** The whole body runs while the `MutexGuard` `g` is held (acquired step 1,
  dropped at scope end), so concurrent `delete`/`create`/`get`/`list` calls cannot observe a
  partially-mutated map. This mirrors Go's whole-body lock (`store.go:77-78`,
  porting-hazards.md H2, H3).

- **Do NOT hold the guard across an `.await`.** This method is fully synchronous; there are no
  await points (porting-hazards.md H2). Use `std::sync::Mutex`, NOT `tokio::sync::Mutex`. `delete`
  itself is a plain (non-`async`) method.

## 4. Acceptance test for THIS slice (must pass)

The Go acceptance test is `TestStoreDelete` (`store_test.go:109-122`). Port it as a Rust unit
test inside a `#[cfg(test)] mod tests` block in the same module as `Store`. It exercises
`new`/`create`/`get` (siblings/kernel — already built) only to set up and verify; this slice does
NOT re-implement them.

```rust
#[test]
fn store_delete_removes_existing_item() {
    let s = Store::new();                       // kernel: empty map, next_id = 1
    let it = s.create("temp").expect("create"); // sibling: it.id == 1 (first id)

    // (a) Deleting an existing id returns Ok.
    s.delete(it.id).expect("delete existing should succeed");

    // (b) After delete, get for the same id returns NotFound (item is gone).
    assert!(
        matches!(s.get(it.id), Err(AppError::NotFound)),
        "after delete, get should be NotFound"
    );

    // (c) Deleting the same (now absent) id again returns NotFound.
    assert!(
        matches!(s.delete(it.id), Err(AppError::NotFound)),
        "repeat delete should be NotFound"
    );
}
```

Mapping to Go test and source:

- (a) success on existing id — `store_test.go:113-115`; produced by Rust step 3+4 (mirrors
  `store.go:82-83`).
- (b) gone afterward — `store_test.go:116-118`; `get` returns `NotFound` because the key was
  removed (`get` mirrors `store.go:53-61`, the removed key path is `store.go:57-58`).
- (c) repeat delete is `NotFound` — `store_test.go:119-121`; produced by Rust step 2 (mirrors
  `store.go:79-80`).

`expect`/`matches!` in TEST code is acceptable (porting-hazards.md H19 permits `.unwrap()`-class
calls in tests). The `matches!` comparison enforces identity-based matching of the error variant
(do NOT assert on a `Display` string).

Run: `cargo test` from the Rust crate root.

- **`Store`** — `struct Store { inner: Mutex<StoreInner>, now: Box<dyn Fn() -> DateTime<Utc> + Send + Sync> }`

- **`Store::create` / `Store::get`** — sibling methods used by the test ONLY for setup/verify.

- **Imports already present at the crate level** (no new imports needed for `delete`):

## 6. Do-NOT list (anti-patterns that break parity)

- Do NOT take `&mut self` — use `&self` with interior mutability (§1).
- Do NOT use `tokio::sync::Mutex` — use `std::sync::Mutex` (§3, porting-hazards.md H2).
- Do NOT make `delete` `async` — it is fully synchronous (§3).
- Do NOT touch `next_id` or `now` inside `delete` (§3).
- Do NOT return a new/allocated error or a string; return the `AppError::NotFound` variant and
  match it by identity (§3, §4).
- Do NOT use `i32`/`usize`/`u64` for the id — use `i64` (§1).
- Do NOT add any `.unwrap()`/`.expect()` other than on `inner.lock()` in non-test code
  (porting-hazards.md H19).
- Do NOT mutate the map on the not-found path (§2 step 2).
