# Rust Slice Hint — 0007 "404 not-found response branch" (EPIC-002-S4)

PRESCRIPTIVE Rust guidance for THIS slice only. One chosen approach — no alternatives.
This is the **not-found branch inside the `GET /items/{id}` axum handler**. It is NOT a
standalone function: it is a conditional in the handler that maps a `NotFound` store error
to an HTTP 404 with body `{"error":"item not found"}`.

Source contract is grounded in:
- `server.go:52` — `it, err := store.Get(id)`
- `server.go:53` — `if errors.Is(err, ErrNotFound)`
- `server.go:54` — `writeError(w, http.StatusNotFound, err.Error())`
- `server.go:55` — `return`
- `server.go:57` — success path `writeJSON(w, http.StatusOK, it)`
- `store.go:13` — `ErrNotFound = errors.New("item not found")`
- `store.go:53-61` — `Store.Get` returns `Item{}, ErrNotFound` on miss (`store.go:57-58`)
- `server.go:82-83` — `writeError` → `writeJSON(w, status, map[string]string{"error": msg})`
- `server.go:76-79` — `writeJSON` sets `Content-Type: application/json`, status, then encodes
- `server_test.go:84-86` — the one existing test: `GET /items/999` → 404 (status only)

## 1. EXACT behavior this slice must reproduce

When the item is NOT found:
1. HTTP status MUST be **404** = `axum::http::StatusCode::NOT_FOUND` (grounded `server.go:54`).
2. Body MUST be exactly `{"error":"item not found"}` (grounded: `err.Error()` at `server.go:54`
   is `ErrNotFound`'s message `item not found` from `store.go:13`; wrapped as
   `{"error": msg}` at `server.go:83`).
3. Response MUST carry header `Content-Type: application/json` (grounded `server.go:77`).
4. The handler MUST return immediately and MUST NOT fall through to the 200 success path
   (`server.go:55` return; success at `server.go:57`).

Trailing newline note (grounded `server.go:79`, `json.NewEncoder(...).Encode` adds `\n`):
the Go body is `{"error":"item not found"}\n`. **In the Rust port, do NOT try to reproduce
the trailing newline.** axum's `Json` responder uses `serde_json::to_vec` which emits NO
trailing newline. The body is `{"error":"item not found"}` (no `\n`). Tests MUST compare with
`trim`/`trim_end` (see §4 AT-2) so the Go-vs-Rust newline difference does not matter. This is
a deliberate, accepted divergence; the JSON value is identical.

Ordering precondition (grounded `server.go:47-50` precedes `server.go:53`): the integer parse
runs first. A non-integer id (`/items/abc`) is **400, never 404** (`server_test.go:87-89`).
This branch fires ONLY after a successful integer parse. Owned by S2; do not alter it.

When the item IS found (contrast, owned by EPIC-002-S3): the `NotFound` arm does not match,
and the handler returns 200 + the item. This slice MUST NOT change that path.

returns a `Response`, which implements `IntoResponse`.

## 3. The exact code for THIS slice

You are editing the `get_one` handler registered for `GET /items/{id}` (from S3). Insert the
not-found arm between the `store.get(id)` call and the success response.

Prescribed handler shape (the not-found branch is the load-bearing part of THIS slice):

```rust
use std::sync::Arc;
use axum::extract::{Path, State};
use axum::http::StatusCode;
use axum::response::{IntoResponse, Response};

// State<Arc<Store>> per dependency-map.md (Arc<Store>, Store holds a Mutex).
async fn get_one(
    State(store): State<Arc<Store>>,
    Path(id_raw): Path<String>,   // Path<String>, NOT Path<i64> — see note below
) -> Response {
    // --- precondition: integer parse (owned by S2; shown for placement only) ---
    let id: i64 = match id_raw.parse::<i64>() {
        Ok(n) => n,
        Err(_) => return write_error(StatusCode::BAD_REQUEST, "id must be an integer"),
    };

    // --- store lookup (owned by S3) ---
    match store.get(id) {
        // === THIS SLICE: the 404 not-found branch ===
        Err(AppError::NotFound) => {
            write_error(StatusCode::NOT_FOUND, AppError::NotFound.to_string())
        }
        // success path (owned by S3) — MUST remain reachable only when not NotFound
        Ok(item) => write_json(StatusCode::OK, item),
        // exhaustiveness: any other AppError → 500 (porting-hazards.md H20).
        // Unreachable for Get (it only returns Ok or NotFound) but required by Rust's
        // exhaustive match. Mirrors Go's defensive 500 elsewhere.
        Err(_) => write_error(StatusCode::INTERNAL_SERVER_ERROR, "internal error"),
    }
}
```

Mapping to the grounded Go branch:
- `if errors.Is(err, ErrNotFound)` (`server.go:53`) → `Err(AppError::NotFound) => { ... }`.
  Rust's `match` on the enum variant is the exact equivalent of `errors.Is` against the
  sentinel here (the store returns the variant directly, mirroring `store.go:58` returning the
  sentinel directly). Do NOT compare strings; match the variant.
- `writeError(w, http.StatusNotFound, err.Error())` (`server.go:54`) →
  `write_error(StatusCode::NOT_FOUND, AppError::NotFound.to_string())`.
  The message MUST be the variant's `Display` (`item not found`, grounded `store.go:13`).
  Do NOT hardcode a different literal; source the string from `AppError::NotFound.to_string()`
  (Go sources it from `err.Error()`).
- `return` (`server.go:55`) → in Rust this `match` arm IS the return value of the function;
  no fall-through occurs because each arm returns a `Response`. The success arm is only taken
  on `Ok`, satisfying the immediate-return guarantee.

### Why `Path<String>` and not `Path<i64>`
Grounded ordering (`server.go:47-50` before `server.go:53`): Go parses the id itself with
`strconv.Atoi` and returns its OWN 400 `{"error":"id must be an integer"}` on failure. If you
used `Path<i64>`, axum would reject a non-integer id with ITS OWN 400 body (different shape)
BEFORE the handler runs — and would never reach the 404 logic. Using `Path<String>` + manual
`parse::<i64>()` preserves both the exact 400 body (S2) and the ordering that makes `/items/abc`
a 400 not a 404 (`server_test.go:87-89`). This matches dependency-map.md item 2 and item 6.
