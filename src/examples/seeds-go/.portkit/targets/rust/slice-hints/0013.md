## 0. What this slice is

Inside the `DELETE /items/{id}` axum handler, parse the `{id}` path segment as an integer. On parse
failure, return HTTP **400** with the exact JSON body `{"error":"id must be an integer"}` and
`Content-Type: application/json`, and return from the handler immediately — **before the store is
ever touched**. On success, the parsed integer flows to `store.delete(id)` (the 404 and 204
branches are owned by other slices: EPIC-004-S5 = 404, EPIC-004-S4 = 204 / slice 0014).

Go source for the guard (`server.go:60-65`):

```go
mux.HandleFunc("DELETE /items/{id}", func(w http.ResponseWriter, r *http.Request) {  // server.go:60
    id, err := strconv.Atoi(r.PathValue("id"))                      // server.go:61
    if err != nil {                                                 // server.go:62
        writeError(w, http.StatusBadRequest, "id must be an integer")  // server.go:63
        return                                                      // server.go:64
    }
    // ... store.Delete(id) / 404 / 204 owned by other slices (server.go:66-71)
})
```

The early `return` at `server.go:64` is **textually and operationally before** the
`store.Delete(...)` call at `server.go:66`. A malformed id therefore NEVER reaches the store.

---

## 1. Hard rules (non-negotiable)

1. **Parse type is `i64`, NOT `i32` / `usize`.** Go `int` is 64-bit on the targets in scope and
   `strconv.Atoi` returns `int` (`porting-hazards.md:166-176`, H8; `server.go:61`). Use
   `.parse::<i64>()`. The `id`, `next_id`, and store key must all be `i64` for consistency with
   sibling slices (`porting-hazards.md:172-176`).

2. **Extract `Path<String>` and parse manually. Do NOT use `Path<i64>`.** axum's `Path<i64>`
   auto-extraction produces its OWN 400 with a DIFFERENT body (a plain-text rejection like
   `Cannot parse "abc" to a i64`), which violates the exact-body contract. You MUST own the parse
   so you own the error response. Grounded: `porting-hazards.md:181-203` (H9) and
   `porting-hazards.md:238-256` (H11).

3. **The error message string MUST be EXACTLY** `id must be an integer` — byte-for-byte. No
   capitalization, no trailing period, no rephrasing (`server.go:63`). This is the SAME literal
   used by the GET guard (`server.go:49`).

4. **The error body MUST serialize to EXACTLY** `{"error":"id must be an integer"}`. In Go this is
   `map[string]string{"error": msg}` (`server.go:83`). Reproduce with the shared error writer (see
   §3). `serde_json` with a single key emits exactly `{"error":"..."}` — single key, so no field
   ordering concerns.

5. **`Content-Type` MUST be `application/json`** on the 400 response (Go: `server.go:77` via
   `writeError` → `writeJSON`). This is provided by the shared `write_json` helper (§3); do not set
   it by hand.

6. **Return immediately after writing the error — BEFORE any store call.** No `store.delete` may
   run on the failure path (`server.go:64` `return`, which precedes `server.go:66`). In Rust this
   is `return write_json(...)` inside the match arm. This ordering is a CRITICAL correctness
   guarantee (neutral doc §2 "Ordering guarantee"; `server.go:64` before `server.go:66`).

7. **No trimming, no special-casing.** `strconv.Atoi` does NOT trim whitespace and accepts only an
   optionally-signed base-10 integer. Rust `i64::parse` has the SAME behavior. A single
   `.parse::<i64>()` + error guard reproduces every edge case automatically. Do NOT add
   `.trim()`, radix handling, or any extra validation (`server.go:61`,
   `porting-hazards.md:188-192`).

## 2. Edge-case behavior (driven by `i64::parse`, which matches `strconv.Atoi`)

These MUST all yield 400 `{"error":"id must be an integer"}` — verified equivalent between
`strconv.Atoi` and `i64::parse` in `porting-hazards.md:188-192`:

| Input `{id}` | Result | Reason |
|---|---|---|
| `"xyz"` | 400 | non-numeric (the only Go-test-covered case, `server_test.go:120`) |
| `"abc"` | 400 | non-numeric |
| `""` (empty) | 400 | empty parses to error |
| `"1.5"` | 400 | float syntax not accepted |
| `"0x1"` | 400 | non-base-10 prefix not accepted |
| `" 1"` / `"1 "` | 400 | leading/trailing whitespace NOT trimmed by either Go or Rust |
| `"99999999999999999999999999999"` (overflow) | 400 | exceeds `i64` range → parse error |
| `"+"` alone | 400 | both Go and Rust error |

These MUST all SUCCEED (parse OK, NO error response, control flows to `store.delete`):

| Input `{id}` | Parsed `i64` | Reason |
|---|---|---|
| `"+5"` | `5` | leading `+` accepted by both |
| `"-5"` | `-5` | leading `-` accepted |
| `"007"` | `7` | leading zeros accepted |
| `"0"` | `0` | accepted |
| `"1"` | `1` | accepted (the success path the 204/404 slices rely on) |

> Do NOT special-case any of these. One `.parse::<i64>()` + the `Err` arm produces all rows above.

> Note on the empty case: a request to `/items/` is matched by the list route `GET /items` only for
> GET; for DELETE there is no `DELETE /items` route, so an empty segment does not normally reach
> this handler (it would 404/405 at the router — `porting-hazards.md:337-360`, H16). Implement the
> guard so that IF an empty string ever reaches the parse it yields 400 (it does automatically).
> Same caveat as the GET twin (`slice-hints/0005.md` §2).

## 3. Dependency: shared error writer (from the SHARED-response-helpers slice)

This slice CALLS the shared writer; it MUST NOT redefine it. The established signature
(`porting-hazards.md:88-96`, H4) is:

```rust
fn write_json<T: Serialize>(status: StatusCode, v: T) -> Response
```

The error envelope in Go is `map[string]string{"error": msg}` (`server.go:83`). Reproduce the
error body with one of these (pick the `json!` form for this slice — fewest moving parts):

```rust
use serde_json::json;
write_json(StatusCode::BAD_REQUEST, json!({ "error": "id must be an integer" }))
```

`write_json` is responsible for setting `Content-Type: application/json` and the status code
(mirrors Go `writeJSON` at `server.go:76-79`). Treat it as a black box: depend only on
status + `{"error":msg}` body + JSON content type.

If your SHARED slice instead exposes a convenience `write_error(status, msg) -> Response` wrapper
(mirroring Go `writeError`, `server.go:82-84`), prefer calling that:

```rust
write_error(StatusCode::BAD_REQUEST, "id must be an integer")
```

Use whichever the SHARED slice actually defined. Do NOT define either here.

## 4. Exact handler code for THIS slice

Handler signature (axum 0.7, manual-parse pattern). `Path<String>` because we parse ourselves:

```rust
use axum::{extract::{Path, State}, http::StatusCode, response::Response};
use serde_json::json;
use std::sync::Arc;

async fn delete_one(
    State(store): State<Arc<Store>>,
    Path(raw_id): Path<String>,
) -> Response {
    // --- THIS SLICE: integer parse + 400 guard (mirrors server.go:61-64) ---
    let id: i64 = match raw_id.parse() {
        Ok(n) => n,
        Err(_) => {
            return write_json(
                StatusCode::BAD_REQUEST,
                json!({ "error": "id must be an integer" }),
            );
        }
    };

    // --- success path: id (i64) flows to store.delete(id) (server.go:66) ---
    // The 404 (Err(NotFound)) and 204 (Ok(())) branches are owned by other slices
    // (EPIC-004-S5 = 404, EPIC-004-S4 = 204; see slice-hints/0014.md). Example shape:
    // match store.delete(id) { Err(AppError::NotFound) => /* 404 slice */, Ok(()) => /* 204 slice */ }
    todo!("store.delete(id) + 404/204 owned by sibling slices (0014/0015)")
}
```

Notes:
- `let id: i64` — the explicit type annotation is what selects `parse::<i64>()`. Do not omit it
  or Rust cannot infer the target type (`porting-hazards.md:172-176`).
- `Err(_)` — discard the parse error; the message is a fixed literal, never `err.to_string()`.
  (Go also uses a fixed literal here, `server.go:63`, NOT `err.Error()`.)
- The `return` inside the `Err` arm is mandatory — it guarantees no store call on failure
  (matches `server.go:64`, which precedes `server.go:66`). This is the §1 rule 6 ordering
  guarantee.
- `State<Arc<Store>>` extractor matches the store-wrapping decision in `dependency-map.md:187-190`
  (`Arc<Store>` containing a `std::sync::Mutex`). The exact `store.delete` API is owned by
  EPIC-004-S1 (`store.go:76-84`); only the parse guard and the `i64` value handed downstream
  belong to THIS slice.

The GET handler (`server.go:46-50`) has the IDENTICAL parse guard (slice 0005). That is a
different slice; the parse logic is the same shape — see the note at the top of this file.

## 6. Tests to ADD (TDD obligation — Go coverage is THIN)

The Go suite tests ONLY `DELETE /items/xyz => 400` (`server_test.go:120-121`, inside
`TestDeleteItemHTTP`). It does NOT assert the JSON body, the Content-Type, other malformed inputs,
or that the store was left untouched. A correct rebuild MUST implement the exact body + header
anyway, and MUST add tests for them.

> Grounding note: the neutral slice doc names a standalone `TestDeleteItem_NonIntegerID_Returns400`
> and cites `server_test.go:120-122`. The ACTUAL source has no such standalone function — the
> assertion lives at `server_test.go:120-121` inside `TestDeleteItemHTTP`. Treat the behavior, not
> the invented function name, as authoritative.

Use the in-process axum test harness with `tower::ServiceExt::oneshot` (the established equivalent
of Go's `httptest` + `ServeHTTP`, `dependency-map.md:237-243`). Dev-dependencies:
`tower = { version = "0.5", features = ["util"] }` and `http-body-util` (`dependency-map.md:243`).
`[UNVERIFIED]` versions — pick latest compatible.

Write ONE `#[tokio::test]` with a `for` loop over cases (Rust has no native subtests;
`dependency-map.md:234-236`). Required assertions:

- **AT-1 (source-verified):** `DELETE /items/xyz` → status `400` (`server_test.go:120-121`).
- **AT-2 (derived, grounded in `server.go:63,77,83`):** status `400` AND body bytes deep-equal
  `{"error":"id must be an integer"}` (parse the body with
  `serde_json::from_slice::<serde_json::Value>` and compare to
  `json!({"error":"id must be an integer"})`).
- **AT-3 (derived, `server.go:77`):** response header `Content-Type` == `application/json`.
- **AT-4 (derived):** at least `"abc"`, `"1.5"`, `"0x1"` → 400 with the same body.
- **AT-5 (derived, ordering — `server.go:64` before `server.go:66`):** with a store containing
  item id 1, `DELETE /items/xyz` returns 400 AND item id 1 still exists afterward (the malformed
  request must not have deleted anything). Verify via a subsequent `GET /items/1` (or a successful
  `DELETE /items/1` → 204).
- **AT-6 (derived):** a valid integer (`DELETE /items/1` after creating an item) → status **!= 400**
  (do NOT assert exactly 204 here; the 204/404 outcome is owned by sibling slices 0014/0015).

Test skeleton:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use axum::body::Body;
    use axum::http::{Request, StatusCode};
    use http_body_util::BodyExt;
    use tower::ServiceExt; // for `oneshot`

    fn app() -> axum::Router {
        new_server(Arc::new(Store::new())) // constructor owned by other slices
    }

    #[tokio::test]
    async fn delete_non_integer_id_returns_400() {
        for raw in ["xyz", "abc", "1.5", "0x1"] {
            let resp = app()
                .oneshot(
                    Request::builder()
                        .method("DELETE")
                        .uri(format!("/items/{raw}"))
                        .body(Body::empty())
                        .unwrap(),
                )
                .await
                .unwrap();

            assert_eq!(resp.status(), StatusCode::BAD_REQUEST, "input {raw:?}");
            assert_eq!(
                resp.headers()
                    .get("content-type")
                    .and_then(|v| v.to_str().ok()),
                Some("application/json"),
                "input {raw:?}",
            );
            let bytes = resp.into_body().collect().await.unwrap().to_bytes();
            let body: serde_json::Value = serde_json::from_slice(&bytes).unwrap();
            assert_eq!(body, serde_json::json!({ "error": "id must be an integer" }), "input {raw:?}");
        }
    }
}
```

Adjust constructor / `Store::new` names to whatever the route + store slices actually defined.

## 7. Anti-patterns — do NOT do these

- ❌ `Path<i64>` (wrong 400 body — `porting-hazards.md:238-256`, H11).
- ❌ `Path<i32>` or `usize` (wrong width — `porting-hazards.md:166-176`, H8).
- ❌ `.unwrap()` / `.expect()` on `parse` (a panic aborts the request; `porting-hazards.md:413-...`, H19).
- ❌ `raw_id.trim().parse()` (Go does not trim; `server.go:61`).
- ❌ Using `err.to_string()` as the message (Go uses a fixed literal, `server.go:63`).
- ❌ Calling `store.delete` (or any store method) on the parse-error path, or before the parse
  (breaks the ordering guarantee; `server.go:64` precedes `server.go:66`).
- ❌ Redefining `write_json` / `write_error` here (owned by SHARED slice, §3).
- ❌ Re-registering the route (owned by the router/EPIC-004-S1 slice, §5).
- ❌ Implementing `store.delete` / the 404 / 204 path (sibling slices 0014/0015,
  `slice-hints/0014.md`).
