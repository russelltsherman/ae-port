# Rust slice hint — 0009 `Store::list` returns all items sorted by ID ascending

Target language: **Rust**. This is the prescriptive build doc for porting the Go `Store.List()`
method (`store.go:64-73`) to Rust. Build exactly what is specified here. Do not invent
alternatives. Every nontrivial claim is grounded to `path:line` in the Go source or to the
already-decided rust target docs (`dependency-map.md`, `porting-hazards.md`).

---

## What this slice is

A single method on the `Store` type that returns **every** stored item as an owned `Vec<Item>`,
sorted **strictly ascending by `id`** (which equals creation order). No errors, no async, no I/O.

Go source: `store.go:64-73`.

---

## Kernel prerequisites (already decided — do NOT redesign here)

These come from the rust target kernel decisions in `porting-hazards.md` / `dependency-map.md`.
Use them exactly; this slice only adds the `list` method.

- **`Item`** — `#[derive(Clone, Serialize, Deserialize)] struct Item` with fields
  `id: i64, name: String, done: bool, created_at: DateTime<Utc>` (renamed via serde `createdAt`).
  Go origin `item.go:13-18`; rust shape decided at `porting-hazards.md:114-115` and id-width
  `i64` decided at `porting-hazards.md:166-175` (Go `int` is 64-bit; `item.go:14`, `store.go:22`).
  `Item` **must** derive `Clone` because you clone values out of the map while holding the lock
  (`porting-hazards.md:24-27`).
- **`Store`** holds its protected state inside a `std::sync::Mutex`, NOT a bare field beside an
  unprotected map. Decided shape (`porting-hazards.md:48-50`):

  ```rust
  struct StoreInner { items: HashMap<i64, Item>, next_id: i64 }
  struct Store { inner: Mutex<StoreInner>, now: Box<dyn Fn() -> DateTime<Utc> + Send + Sync> }
  ```

  Use `std::sync::Mutex`, **never** `tokio::sync::Mutex` — the critical section has no `.await`
  (`porting-hazards.md:56-57`). Maps to Go `store.go:19-24`.
- **`items` is a `HashMap<i64, Item>`** (`porting-hazards.md:28`), mirroring Go's
  `map[int]Item` (`store.go:21`). Like Go maps, Rust `HashMap` iteration order is unordered /
  randomized — the sort is what makes output deterministic (`porting-hazards.md:467`).
- **`next_id` starts at 1** (`porting-hazards.md:52`; Go `store.go:28`). Not load-bearing for
  `list` itself, but it is why ascending-by-id == creation order.

This slice's implementation depends ONLY on the `StoreInner`/`Item` shapes above. It does not
depend on the Create slice's code (only on `items` being populated). Same as Go note in the
neutral doc.

## EXACT signature to build

```rust
impl Store {
    /// Returns all items sorted by id ascending (creation order).
    pub fn list(&self) -> Vec<Item> {
        // ...
    }
}
```

- Receiver: `&self` (shared borrow — listing does not mutate the `Store`; it locks internally).
  Mirrors Go value-semantics method `func (s *Store) List() []Item` (`store.go:64`).
- **No parameters.**
- **Return type: `Vec<Item>`** — owned, by-value. NOT `&[Item]`, NOT `Vec<&Item>`, NOT
  `Result<Vec<Item>, _>`, NOT `Option<Vec<Item>>`.
  - It returns owned clones (not references) because the borrowed data lives behind the `Mutex`
    guard, which drops at end of function — you cannot return a reference into it
    (`porting-hazards.md:19-20`, `24-27`).
  - It is `Vec<Item>` not `Result<...>` because Go `List` has **no error path** and cannot fail
    (`store.go:64-73`; neutral doc §"No errors"; `porting-hazards.md:331-332`).
  - It is `Vec<Item>` not `Option<Vec<Item>>`: an empty store returns an **empty `Vec`** (`vec![]`
    / length 0), which is the non-`null` analog of Go's non-nil empty slice
    (`porting-hazards.md:478-488`; Go `store.go:67`).

## EXACT method body (build this)

```rust
pub fn list(&self) -> Vec<Item> {
    let g = self.inner.lock().unwrap();
    let mut out: Vec<Item> = Vec::with_capacity(g.items.len());
    for it in g.items.values() {
        out.push(it.clone());
    }
    out.sort_by_key(|it| it.id);
    out
}
```

Line-by-line, with Go grounding:

1. **Acquire the lock for the whole read.** `let g = self.inner.lock().unwrap();`
   - Mirrors Go `s.mu.Lock()` + `defer s.mu.Unlock()` (`store.go:65-66`). In Rust the RAII guard
     `g` is held until the end of the function (when it drops), so the entire collect-and-sort
     happens under the lock — same as Go (`porting-hazards.md:32-53`).
   - `.unwrap()` on `lock()` is the ONE sanctioned unwrap: poisoning is unreachable in correct
     code; Go has no poisoned-lock concept (`porting-hazards.md:44`, `427`). Do not propagate it
     as an error — `list` has no error return.

2. **Allocate the result with a capacity hint.**
   `let mut out: Vec<Item> = Vec::with_capacity(g.items.len());`
   - Mirrors Go `out := make([]Item, 0, len(s.items))` (`store.go:67`). Length 0, capacity =
     number of stored items. The capacity hint is perf-only and optional
     (`porting-hazards.md:473`), but build it as shown for fidelity. An empty store yields an
     empty (length-0) `Vec` — this is correct and is the `[]`-serializing value
     (`porting-hazards.md:478-488`).

3. **Collect cloned values from the map.**
   `for it in g.items.values() { out.push(it.clone()); }`
   - Mirrors Go `for _, it := range s.items { out = append(out, it) }` (`store.go:68-70`).
   - Iterate **values** (`g.items.values()`), not entries — you don't need the key.
   - Push `it.clone()` (an owned `Item`), reproducing Go's copy-out value semantics
     (`porting-hazards.md:24-27`). Do NOT push `&Item` references.
   - Order here is unspecified (HashMap iteration is randomized) — the next step fixes that
     (`porting-hazards.md:467`).

4. **Sort strictly ascending by id.** `out.sort_by_key(|it| it.id);`
   - Mirrors Go `sort.Slice(out, func(i, j int) bool { return out[i].ID < out[j].ID })`
     (`store.go:71`). `sort_by_key` on the `i64` `id` field gives ascending order.
   - IDs are unique map keys (Go `store.go:21`; rust `HashMap` key), so there are never ties:
     stable vs unstable sort is irrelevant here (`porting-hazards.md:472`). `sort_by_key`
     (stable) is fine; do not bother with `sort_unstable_by_key`.
   - Do NOT skip this. Without it, output order is nondeterministic and the ordering test fails
     (`porting-hazards.md:467-469`).

5. **Return the vec.** `out` (last expression). Mirrors Go `return out` (`store.go:72`). The guard
   `g` drops here, releasing the lock.

## EXACT contract (behavior the rebuilt code MUST satisfy)

Grounded in the neutral slice doc and Go source.

1. **Length.** `list().len()` == number of items in `items` (`g.items.len()`). One element per
   stored item; no duplicates, no omissions. (Go `store.go:67-70`.)
2. **Strict ascending total order by id.** For every adjacent pair, `out[i-1].id < out[i].id`
   (strict `<`, never `<=`). Total order with no ties because ids are unique
   (Go `store.go:71`, `21`).
3. **Equivalence to creation order.** ids are assigned 1,2,3,… in creation order, so ascending-id
   == creation order (Go `store.go:28,46,48`; `next_id` starts at 1 per
   `porting-hazards.md:52`).
4. **Empty store → empty, non-`null` vec.** Zero items ⇒ `Vec` of length 0. Serializes to `[]`
   via serde_json, never `null` (`porting-hazards.md:478-488`; analog of Go non-nil slice
   `store.go:67`). `[UNVERIFIED BY TEST]` — no existing test exercises the empty case; this is a
   source/contract fact, not test-backed.
5. **Snapshot / copy semantics.** Returns owned `Item` clones; mutating the returned `Vec` does
   not affect the store. Contents reflect store state at lock-hold time
   (`porting-hazards.md:24-27`; Go `store.go:68-69`).
6. **Thread safety.** Entire collect+sort runs while holding `inner.lock()` — safe concurrently
   with other `Store` methods that also lock `inner` (`porting-hazards.md:32-53`; Go
   `store.go:65-66`).
7. **No errors, no panic.** Returns `Vec<Item>` directly; cannot fail; does not panic for empty
   or populated stores (the only `.unwrap()` is the unreachable poison case). (Go `store.go:64-73`
   has no error path.)
8. **ID reuse after deletion.** ids are never reused (`next_id` only increments), so after
   deletes + re-creates the result is still strictly ascending with possible gaps.
   `[UNVERIFIED BY TEST]` — grounded in source only (Go `store.go:48`; delete does not decrement,
   `store.go:82`).

### Test 1 — populated store: count and strict ascending order (test-backed)

Ports Go `store_test.go:91-107`.

```rust
#[test]
fn store_list_is_sorted_by_id() {
    let s = Store::new();
    for n in ["a", "b", "c"] {
        s.create(n).expect("create");
    }
    let list = s.list();
    assert_eq!(list.len(), 3, "list len");
    for i in 1..list.len() {
        assert!(
            list[i - 1].id < list[i].id,

            list.iter().map(|it| it.id).collect::<Vec<_>>()

}
```

- Concrete grounded expectation: ids are `[1, 2, 3]` (first create → id 1 per
  `store_test.go:38`; second → id 2 per `store_test.go:50-51`; third → 3). The Go test does NOT
  assert names, only count + id ordering — do not add name assertions here.
- `s.create(...)` signature/return is owned by the Create slice; if it returns
  `Result<Item, StoreError>` use `.expect("create")` as shown. Adjust only the error-handling

## Build steps (each individually checkable)

1. **Confirm imports / deps.** `std::collections::HashMap` and `std::sync::Mutex` are in scope on
   the `Store` module (kernel already needs them). `Vec::with_capacity` and `sort_by_key` are std,
   no extra imports. (No `sort` crate — Go's `sort` import at `store.go:5` maps to the std slice
   method, `dependency-map.md:175`.) *Check:* `cargo build` clean.
2. **Add the method** in `impl Store` with the exact signature above and a doc comment
   `/// Returns all items sorted by id ascending (creation order).` (maps Go doc `store.go:63`).
   *Check:* compiles with a `todo!()` body.
3. **Lock for the duration:** `let g = self.inner.lock().unwrap();` as the first line.
   *Check:* `cargo clippy` clean; no `tokio::Mutex`.
4. **Allocate** `let mut out: Vec<Item> = Vec::with_capacity(g.items.len());`. Use
   `with_capacity`, not `Vec::new()` for fidelity (`store.go:67`). *Check:* empty store → empty
   vec (Test 2).
5. **Collect** `for it in g.items.values() { out.push(it.clone()); }`. Clone values; iterate
   `.values()`. *Check:* 3-item store → `out.len() == 3` before sort.
6. **Sort** `out.sort_by_key(|it| it.id);` (ascending). *Check:* Test 1's ascending assertion.
7. **Return** `out`. *Check:* both tests pass.
8. **Run** `cargo test` (and `cargo test` under the thread sanitizer is N/A; Rust's type system +
   `Send + Sync` on `Store` per `porting-hazards.md:75` cover the race concern). *Check:*
   `store_list_is_sorted_by_id` and `store_list_empty_is_empty_vec` pass.

## Pitfalls (do NOT do these)

- Do NOT return `Result<Vec<Item>, _>` — `list` cannot fail (`store.go:64-73`).
- Do NOT return `Option<Vec<Item>>` — empty must be `[]`, an `Option::None` serializes to `null`
  (`porting-hazards.md:485-486`).
- Do NOT return references (`Vec<&Item>` / `&[Item]`) — the data is behind a `Mutex` guard that
  drops on return (`porting-hazards.md:19-20`).
- Do NOT use `tokio::sync::Mutex` (`porting-hazards.md:56-57`).
- Do NOT skip `sort_by_key` and rely on `HashMap` iteration order — it is randomized
  (`porting-hazards.md:467`).
- Do NOT use `<=` in the sort/comparison — ordering is strict `<` (Go `store.go:71`).
- Do NOT `.unwrap()` anything except `inner.lock()` (`porting-hazards.md:427`).
