# Rust slice hint — #10 · GET /items route returns 200 with the JSON item list (EPIC-003-S3)

PRESCRIPTIVE Rust guidance for THIS slice only. The route handler is one line in Go
(`server.go:43`); the work here is wiring it into `axum` and preserving four exact contracts:
status 200, `Content-Type: application/json`, ascending-by-id order, and empty store → `[]`
(never `null`).

Stack is FIXED by `dependency-map.md`: `axum` 0.7 on `tokio`, `serde` + `serde_json`, `chrono`.
Do not substitute. This slice consumes two prerequisites as black boxes — do NOT reimplement
them here:
- `Store::list()` (from EPIC-003-S1) — returns the sorted `Vec<Item>`.
- the JSON 200 responder (from SHARED-response-helpers) — serializes with the right header/status.

---

## 1. Exact API to use

### Router registration (axum 0.7)
Register the GET handler on the `/items` route. In this codebase `/items` ALSO carries `POST`
(`server.go:21,42`), so the two methods are chained on ONE route entry
(porting-hazards.md:351-354):

```rust
use axum::routing::{get, post};
use axum::Router;

Router::new()
    .route("/items", post(create_item).get(list_items))
    // ... other routes ...
    .with_state(store)
```

For THIS slice you are adding the `.get(list_items)` part. The path string is EXACTLY `"/items"`
— mirrors the Go pattern `"GET /items"` (`server.go:42`) minus the method prefix (axum encodes
the method via `get(...)`, not in the path string).

> axum 0.7 uses method-router combinators; `post(h).get(h2)` registers both on the same path and
> yields a 405 automatically for other methods (porting-hazards.md:344-346). No test asserts the
> 405 body, so the axum default is acceptable.

### Handler signature
The Go handler ignores `r` entirely — it reads no body, no query, no path params
(slice doc §2; `server.go:43`). The Rust handler therefore takes ONLY the shared state and
nothing else. Use the EXACT signature:

```rust
use std::sync::Arc;
use axum::extract::State;
use axum::Json;

async fn list_items(State(store): State<Arc<Store>>) -> Json<Vec<Item>> {
    Json(store.list())
}
```

- `async fn` — required: axum handlers are async (the runtime is `tokio`, dependency-map.md:24).
- `State(store): State<Arc<Store>>` — the store is shared across handlers via `Arc<Store>`
  (dependency-map.md:190; porting-hazards.md:354 `.with_state(store)`). Do NOT take any other
  extractor; adding `Json<...>` input, `Query`, or `Path` here is WRONG — the handler MUST NOT
  parse or branch on the request (slice doc §2 "Inputs consumed by the handler: NONE").
- Return `Json<Vec<Item>>`. This is the SHARED-response-helpers contract realized via axum's
  `Json` responder, which sets `Content-Type: application/json` automatically
  (porting-hazards.md:375-376) and a default `200 OK` status (matching `http.StatusOK` at
  `server.go:43`). Because 200 is axum's default for a bare `Json<T>`, you do NOT need to wrap it
  in a tuple `(StatusCode::OK, Json(...))` — but doing so is equally correct and explicit.

The body is `Json(store.list())` and NOTHING else — no `.clone()`, no sort, no filter, no
post-processing (slice doc §5 step 3,5; `server.go:43`). `Store::list()` already returns the
fully-sorted vec.

## 2. The four contracts and how each is satisfied

| Contract | Source fact | Rust mechanism |
|---|---|---|
| Status `200 OK` | `server.go:43` (`http.StatusOK`); slice §2 "the only outcome" | `Json<Vec<Item>>` default status is 200. No error branch exists — do NOT return `Result` here. |
| `Content-Type: application/json` | `server.go:77`; slice AT-5 | axum `Json` responder sets it automatically (porting-hazards.md:375-376). |
| Elements ordered by `id` ascending | `store.go:71` (`out[i].ID < out[j].ID`); slice §2, AT-4 | Provided by `Store::list()` (`out.sort_by_key(|it| it.id)`, porting-hazards.md:471-472). Do NOT re-sort here. |
| Empty store → `[]` not `null` | `store.go:67` non-nil empty slice; slice §2, AT-3 | `Store::list()` returns `Vec<Item>` (not `Option<Vec<_>>`). An empty `Vec` serializes to `[]` via serde_json by default (porting-hazards.md:484-488). |

### Critical: never `Option<Vec<Item>>`
The single most likely way to BREAK this slice in Rust: returning `Option<Vec<Item>>` or
otherwise letting an empty result become `null`. `None` / `null` VIOLATES the contract
(slice §2 empty-store guarantee; porting-hazards.md:485-486). `Store::list()` returns a plain
`Vec<Item>`; an empty `Vec` is `[]`. Keep the type `Vec<Item>` end-to-end. This is the AT-3 gap
that passes no existing test but is a hard contract.

### `Item` serialized shape (relied upon, defined by prerequisite slices)
`Item` is a `#[derive(Serialize)]` struct with serde rename attributes matching the Go tags
(`item.go:13-18`; dependency-map.md:204-209):
- `id` → integer (`item.go:14`)
- `name` → string (`item.go:15`)
- `done` → bool (`item.go:16`)
- `createdAt` → RFC 3339 string, `chrono::DateTime<Utc>` serde-renamed to `createdAt`
  (`item.go:17`; dependency-map.md:206-209). The timestamp format divergence is a known
  HUMAN-DECISION (dependency-map.md:210-221) but is OUT OF SCOPE for this slice — no test here
  asserts `createdAt` content.

Do not redefine `Item` in this slice; it is owned by EPIC-003-S1. Listed here only so the
rebuilder knows what the array elements look like.

## 4. Acceptance tests (Rust)

Use the in-process `tower::ServiceExt::oneshot` harness — the idiomatic axum equivalent of Go's
`srv.ServeHTTP(recorder, req)` (dependency-map.md:239-242). Dev-dependencies required:
`tower = { version = "0.5", features = ["util"] }`, `http-body-util`, plus `serde_json` for body
assertions. Build the router via the project's constructor (e.g. `build_router(Arc::new(Store::new()))`,
porting-hazards.md:396) and seed items by issuing real `POST /items` requests, exactly as the Go
test does (`server_test.go:94-95`).

Helper pattern (collect the body bytes):

```rust
use tower::ServiceExt; // for .oneshot
use http_body_util::BodyExt; // for .collect().to_bytes()
use axum::http::{Request, StatusCode};
use axum::body::Body;

async fn send(app: &Router, method: &str, path: &str, body: &str) -> (StatusCode, Vec<u8>) {
    let req = Request::builder()
        .method(method)
        .uri(path)
        .header("content-type", "application/json")
        .body(if body.is_empty() { Body::empty() } else { Body::from(body.to_string()) })
        .unwrap();
    let resp = app.clone().oneshot(req).await.unwrap();
    let status = resp.status();
    let bytes = resp.into_body().collect().await.unwrap().to_bytes().to_vec();
    (status, bytes)
}
```

### AT-1 — Two items, status 200 (mirrors `server_test.go:92-99`)
Create `{"name":"a"}` then `{"name":"b"}` via POST, then `GET /items`; assert status `200`.
```rust
assert_eq!(status, StatusCode::OK);
```

### AT-2 — Body is a JSON array of length 2 (mirrors `server_test.go:101-107`)
Decode the body into `Vec<Item>` and assert length 2 (NOT an object, NOT a single item).
```rust
let items: Vec<Item> = serde_json::from_slice(&bytes).unwrap();
assert_eq!(items.len(), 2);
```

### AT-3 — Empty store returns `[]`, not `null` (NEW — fills the LOUD gap, slice AT-3)
Fresh store, no items, `GET /items`. Assert status 200 AND the raw body trims to exactly `[]`.
```rust
assert_eq!(status, StatusCode::OK);
assert_eq!(std::str::from_utf8(&bytes).unwrap().trim(), "[]");
```
This is the contract from `store.go:67` / porting-hazards.md:484-488. No Go test covers it.
A handler returning `null` would still pass AT-1/AT-2 but is WRONG.

### AT-4 — Ascending-by-id order at the HTTP layer (NEW — slice AT-4)
Create `a`, `b`, `c` (ids 1,2,3 per `store.go:28,46-48`), `GET /items`, assert strictly
increasing ids.
```rust
let items: Vec<Item> = serde_json::from_slice(&bytes).unwrap();
assert!(items[0].id < items[1].id && items[1].id < items[2].id);
```

### AT-5 — Content-Type is application/json (NEW — slice AT-5)
Assert the response header `Content-Type` equals `application/json`. axum's `Json` responder
emits `application/json` (porting-hazards.md:375). Note: axum may append `; charset=utf-8` —
the Go source sets the bare string `"application/json"` (`server.go:77`). If exact byte-match
matters, assert with `starts_with("application/json")`; the Go value has no charset suffix.
```rust
let ct = resp.headers().get("content-type").unwrap().to_str().unwrap();
assert!(ct.starts_with("application/json"));
```
> `[UNVERIFIED]` — whether the downstream requires the exact bare `application/json` (no
> `; charset=utf-8`). The Go source emits no charset (`server.go:77`). If strict equality is
> required, build the response manually setting the header to exactly `"application/json"`
> instead of relying on axum's `Json` default. Otherwise `starts_with` is the pragmatic check.

## 5. Do-NOT list (anti-patterns that pass weak tests but break the contract)

- Do NOT return `Option<Vec<Item>>` / anything that can serialize to `null` (AT-3;
  porting-hazards.md:485-486).
- Do NOT sort, filter, or post-process inside the handler — `Store::list()` owns ordering
  (`store.go:71`; slice §5 step 5).
- Do NOT add a `Result` return, error branch, status other than 200, or any request extractor
  (`server.go:42-44`; slice §2 error-behavior).
- Do NOT inline `serde_json::to_string` / manual encoding — use the `Json` responder
  (SHARED-response-helpers contract; slice §3).
- Do NOT change the route path string from `"/items"` or split it into a separate route from the
  existing `POST /items` — chain `.get` onto the same route (porting-hazards.md:352).
