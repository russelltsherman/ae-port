## 0. What this slice owns (and what it does NOT)

This slice owns EXACTLY two things (from the slice doc §1, grounded in `server.go`):

1. **Register the single-item GET route** with the exact method+path pattern `GET /items/{id}`
   inside the server constructor (`server.go:18`, `server.go:19`, `server.go:46`).
2. **Extract the `{id}` path segment as a raw string** inside that handler via
   `r.PathValue("id")` (`server.go:47`), and hand it OFF to the parser **unaltered** — no
   lowercasing, trimming, or any transformation. Go passes it directly into `strconv.Atoi`
   on the same line (`server.go:47`).

This slice does NOT own: the integer parse, the store lookup, the success/error response bodies,
or the other three routes' handler bodies. Those belong to other EPIC-002 slices. You will,
however, have to stub the rest of the handler so the code compiles and so the acceptance tests
(which exercise the full handler) pass — see §5.

## 2. The constructor and router — exact shape

Go's `func NewServer(store *Store) http.Handler` (`server.go:18`) creates a `*http.ServeMux`
(`server.go:19`), registers four routes, and returns the mux (`server.go:73`). For Rust, the
1:1 equivalent is a function that builds and returns an `axum::Router`.

**Use this exact constructor signature and name** (mirrors `NewServer`):

```rust
use std::sync::Arc;
use axum::{Router, routing::{get, post, delete}};

pub fn new_server(store: Arc<Store>) -> Router {
    Router::new()
        .route("/items", post(create_item).get(list_items))
        .route("/items/{id}", get(get_item).delete(delete_item))
        .with_state(store)
}
```

Grounding for each route (so the router matches the Go mux exactly):

- `post("/items")` ⇐ `mux.HandleFunc("POST /items", ...)` (`server.go:21`).
- `get("/items")` ⇐ `mux.HandleFunc("GET /items", ...)` (`server.go:42`).
- **`get("/items/{id}")` ⇐ `mux.HandleFunc("GET /items/{id}", ...)` (`server.go:46`) — THIS SLICE.**
- `delete("/items/{id}")` ⇐ `mux.HandleFunc("DELETE /items/{id}", ...)` (`server.go:60`).

Notes that are load-bearing for correctness:

- **Two methods on the SAME path string share one `.route(...)` call.** `GET /items` and
  `POST /items` both go on `.route("/items", post(...).get(...))`. `GET /items/{id}` and
  `DELETE /items/{id}` both go on `.route("/items/{id}", get(...).delete(...))`. This reproduces
  Go's method-specific dispatch where each `(method, path)` pair is a distinct handler
  (`server.go:46` GET vs `server.go:60` DELETE are independent — slice doc §3.3).
- **`{id}` is a single-segment wildcard.** It matches exactly ONE non-empty path segment and does
  NOT cross `/` (slice doc §3.2). axum's `{id}` has the same single-segment semantics; do NOT use
  a wildcard-capture form (`{*id}`).
- **`/items` (no segment) must NOT fall into the `{id}` route.** axum distinguishes the two paths
  structurally, so `GET /items` hits `list_items` and `GET /items/1` hits `get_item`. This matches
  Go's `ServeMux` specificity (slice doc §3.3, `server.go:42` vs `server.go:46`). Do NOT register
  `{id}` in any way that also matches the bare `/items` path.
- `with_state(store)` makes the shared `Arc<Store>` available to handlers (Go closes over the
  single `*Store` — `server.go:18`). The store type/constructor are KERNEL (`store.go:19-29`); do
  not redefine them in this slice.
- **Return type is `Router`** (the slice doc §3.1 maps Go's `http.Handler` return to an
  `axum::Router`; `server.go:73` returns the mux). Do not wrap it in `Result` or `Box`.

## 3. THE CORE OF THIS SLICE — path id extraction

Go reads the wildcard with `r.PathValue("id")` and passes it straight into `strconv.Atoi` with no
transformation, on a single line (`server.go:47`):

```go
id, err := strconv.Atoi(r.PathValue("id"))
```

### 3.1 How to extract the raw segment in axum

Use the `Path<String>` extractor — **NOT** `Path<i64>`. Extracting as `String` gives you the raw
segment exactly as Go's `PathValue("id")` does (a `string`), so you control the parse and the
error body yourself. (Using `Path<i64>` would let axum parse and emit its OWN 400 body, diverging
from Go — see `.portkit/targets/rust/porting-hazards.md` H9/H11. That parse + body belongs to a
downstream slice anyway, but the EXTRACTION shape this slice establishes must be `String`.)

```rust
use axum::extract::{Path, State};
use axum::response::Response;

async fn get_item(
    State(store): State<Arc<Store>>,
    Path(id): Path<String>,           // <-- raw segment, exactly like r.PathValue("id")
) -> Response {
    // `id` is the raw matched segment, URL-path-decoded by the router,
    // handed off UNALTERED to the downstream parse (other slice).
    // ... downstream parse + lookup live in another slice; see §5 stub ...
}
```

Grounding and exact-behavior requirements:

- The extractor name binding is `Path(id)` and the type is `Path<String>`. The capture name in the
  route pattern (`{id}`) and the extractor must agree on the segment. In Go the agreement is the
  literal string argument `"id"` matching `{id}` (`server.go:46` pattern vs `server.go:47`
  `PathValue("id")`); a mismatch in Go returns `""` (slice doc §3.4). In axum the single-segment
  `{id}` binds to a single-field `Path<String>` positionally — keep the field/route name `id` for
  fidelity.
- **The returned string is URL-path-decoded.** Go's `PathValue` returns the segment decoded per
  `net/http` (slice doc §3.4: `/items/1` → `"1"`, `/items/abc` → `"abc"`). axum's `Path<String>`
  also returns the percent-decoded segment. Behavior matches for the ASCII inputs the tests use.
- **DO NOT TRANSFORM `id`.** No `.to_lowercase()`, `.to_uppercase()`, `.trim()`, character
  stripping, or normalization before it reaches the downstream parser. Go passes it verbatim into
  `strconv.Atoi` (`server.go:47`; slice doc §3.4, §6 step 6). Case must be preserved exactly:
  `/items/AbC` must yield the raw string `"AbC"`. Any transformation is OUT of contract.

### 3.2 Handler signatures for the other three routes (stubs this slice provides)

So the router compiles, declare these with matching extractor shapes (their bodies are other
slices' work — stub per §5):

```rust
async fn create_item(State(store): State<Arc<Store>>, body: axum::body::Bytes) -> Response;
async fn list_items (State(store): State<Arc<Store>>) -> Response;
async fn get_item   (State(store): State<Arc<Store>>, Path(id): Path<String>) -> Response;
async fn delete_item(State(store): State<Arc<Store>>, Path(id): Path<String>) -> Response;
```

- `create_item` takes `Bytes` (raw body) — NOT `Json<T>` — so a downstream slice can reproduce
  Go's manual decode + `{"error":"invalid JSON body"}` on failure (`server.go:25-27`;
  porting-hazards H11). This slice only needs the signature to exist.
- `get_item` and `delete_item` both take `Path<String>` because both Go routes read the raw
  segment the same way (`server.go:47`, `server.go:61`).

## 4. Extractor ordering rule (axum-specific gotcha)

In axum, `State<T>` and `Path<T>` can appear in any order EXCEPT that any body-consuming extractor
(`Bytes`, `String`, `Json<T>`) must be **last** in the argument list. The signatures in §3.2 already
follow this (`Bytes` is last in `create_item`). For `get_item`/`delete_item` there is no body
extractor, so `State` then `Path` is fine. Do not put a second body-consuming extractor anywhere.

## 5. Minimal stubs to make it compile + pass the acceptance tests

The acceptance tests exercise the FULL handler (status 200/404/400 — slice doc §5). To get the
route + extraction observable, you must also wire the trivial downstream so the handler returns
something. Since the parse/lookup/response bodies are other slices, the smallest correct stub for
THIS slice's tests is to do the parse + lookup inline; mark these lines as borrowed from downstream
slices and replace them when those slices land.

```rust
use axum::http::StatusCode;
use axum::Json;
use serde_json::json;

async fn get_item(
    State(store): State<Arc<Store>>,
    Path(id): Path<String>,
) -> Response {
    // --- THIS SLICE: raw segment extracted above as `id` (String), untransformed ---

    // --- downstream slice (parse): Go server.go:47-50 ---
    let id: i64 = match id.parse() {
        Ok(v) => v,
        Err(_) => return (
            StatusCode::BAD_REQUEST,
            Json(json!({ "error": "id must be an integer" })),
        ).into_response(),
    };

    // --- downstream slice (lookup + mapping): Go server.go:52-57 ---
    match store.get(id) {
        Ok(item) => (StatusCode::OK, Json(item)).into_response(),
        Err(_)   => (
            StatusCode::NOT_FOUND,
            Json(json!({ "error": "item not found" })),
        ).into_response(),
    }
}
```

(`store.get` is the kernel `*Store`'s `Get` — `store.go:53`; `ErrNotFound` message is
`"item not found"`, `store.go:13`. These belong to other slices; included only so this slice's
route is observable.) Use `axum::response::IntoResponse` (`.into_response()`) so the match arms
have a single `Response` type.

For the other handlers, provide whatever minimal stub the test harness needs (e.g. `create_item`
parsing the body and creating; `delete_item` parsing + deleting; `list_items` returning the list).
Those are other slices' deliverables — keep them minimal here and replace later.

Test helper (Rust analog of `newTestServer` / `do`, `server_test.go:11-26`):

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use axum::body::Body;
    use axum::http::{Request, StatusCode};
    use http_body_util::BodyExt;
    use tower::ServiceExt; // for `oneshot`

    fn test_server() -> Router { new_server(Arc::new(Store::new())) }

    async fn do_req(app: Router, method: &str, path: &str, body: &str) -> (StatusCode, String) {
        let req = Request::builder()
            .method(method)
            .uri(path)
            .header("content-type", "application/json")
            .body(Body::from(body.to_owned()))
            .unwrap();
        let resp = app.oneshot(req).await.unwrap();
        let status = resp.status();
        let bytes = resp.into_body().collect().await.unwrap().to_bytes();
        (status, String::from_utf8(bytes.to_vec()).unwrap())
    }
```

- **AT-1 — route reached, valid id → 200** (Go `server_test.go:75-82`): POST `/items`
  `{"name":"findme"}` to create id 1, then GET `/items/1` ⇒ status `200`. Proves
  `GET /items/{id}` matches and `id="1"` is read.
- **AT-2 — non-numeric segment reaches handler → 400** (Go `server_test.go:87-89`): GET
  `/items/abc` ⇒ status `400`. A `400` (not `404`) proves the route was registered AND the segment
  was captured — if the route were missing, axum would return `404`.
- **AT-3 — numeric-but-missing id reaches handler → 404** (Go `server_test.go:84-86`): GET
  `/items/999` ⇒ status `404`. Proves the segment is captured and parsed (route reached), then the
  store lookup misses.
- **AT-4 — method specificity** (slice doc §5, Go `server_test.go:114-116`): POST `/items`
  `{"name":"temp"}`, then DELETE `/items/1` ⇒ status `204`. Proves DELETE on the same path shape is

- **AT-5 — list route vs single-item route do not collide** (slice doc §5, Go
  `server_test.go:97-100`): POST `/items` `{"name":"a"}`, then GET `/items` ⇒ status `200` from the

- **AT-6 — direct extraction proof (optional, NOT in Go source; slice doc §5 NOTE).** To directly

## 8. Hard "do NOT" list (fidelity guardrails)

- Do NOT use `Path<i64>` for extraction in this slice — use `Path<String>` (raw segment, like
  `r.PathValue("id")`). axum auto-parse would change the 400 body (porting-hazards H9/H11).
- Do NOT transform the extracted string (no lower/upper/trim/strip) before the downstream parse
  (`server.go:47`; slice doc §3.4).
- Do NOT collapse GET and DELETE on `/items/{id}` into one handler — register both methods
  separately on the same `.route` (`server.go:46` vs `server.go:60`).
- Do NOT register `{id}` so it also matches the bare `/items` (keep list vs single-item distinct;
  `server.go:42` vs `server.go:46`).
- Do NOT use a multi-segment wildcard (`{*id}` / `{id...}`); `{id}` is single-segment
  (slice doc §3.2).
- Do NOT change the constructor return type away from `Router`, and do NOT make it fallible
  (`server.go:73` returns the mux directly).
